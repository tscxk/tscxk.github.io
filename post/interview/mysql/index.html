Mysql基础

1. MySQL数据库中MyISAM和InnoDB的区别

MyISAM:

  • 不支持事务，但每次查询都是原子的
  • 不支持行级锁
  • 不支持外键约束
  • 使用非聚簇索引
  • 存储表的总行数

InnoDB:

  • 支持事务并且有四种隔离级别
  • 支持行级锁
  • 拥有redo log，支持崩溃后数据恢复
  • 用户undo log，支持事务回滚
  • 使用聚簇索引
  • 不存储表的总行数

2. 数据库的三范式

  1. 每个字段具有原子性，即每列数据不可再拆分
  2. 非主键字段完全依赖主键
  3. 非主键字段直接依赖主键字段，不存在间接依赖

3. B树和B+树 为什么Mysql选择B+树

B树和B+树都是多路平衡搜索树，B+树是B树的变种，在数据库索引领域几乎取代了B树

  1. B树结构特点：

    • 所有节点存储数据
    • 叶子结点和非叶子节点结构相同，非叶子节点的键不会在叶子节点中再次出现
    • 节点内部存储数据有序：节点内部数据可以用二分法查找

  2. B+树结构特点：

    • 数据只存储在叶子节点，非叶子节点只存储指针
    • 叶子节点形成有序链表
    • 非叶子节点的键会在叶子结点再次出现

  3. 为什么选择B+树

    • 更低的树高，更少的IO

      由于非叶子节点不存储数据，这意味着每个非叶子节点可以存储更多的键，在相同的数据规模的情况下，B+树会更矮，这会消耗更少的IO次数

    • 支持范围查询

      由于B+数的叶子节点有序，这意味着它可以更好的支持范围查询，相较之下，B树则需要复杂的中序遍历

    • 更稳定的查询效率

      这是B+树相对于B树最大的优势。叶子节点的链表结构使得范围查询几乎是线性的，性能极佳。而B树需要进行复杂的中序遍历。

    • 更好的空间局部性

      叶子节点存储所有数据，并且是顺序的，更适合磁盘的预读特性。全表扫描只需要遍历叶子节点链表即可，效率非常高。

4. 一条sql的执行过程

  1. 连接器

    作用：管理客户端连接、身份认证和权限校验

  2. 查询缓存

    作用：缓存select查询结果，如果命中则直接缓存，否则继续执行并将结果存入缓存

    但是在Mysql8.0之后，查询缓存已被移除

    原因：任何对表的更改操作（INSERT/UPDATE/DELETE）都会使该表的所有查询缓存失效。对于更新频繁的系统，缓存命中率极低，维护缓存反而带来巨大开销。

  3. 分析器

    作用：进行词法分析+语法分析

    • 词法分析：对sql语句进行拆分识别，例如识别出字段、表、查询条件等
    • 语法分析：根据mysql的语法规则判断SQL语法的正确性

  4. 优化器

    作用：它会选择它认为成本最低的方案，但这个方案不一定总是最优的。这正是有时需要使用FORCE INDEX或优化SQL写法来干预的原因。它本身主要关注以下几个方面：

    • 索引选择
    • 多表关联的顺序
    • 子查询优化
    • 条件排序和化简

    执行器主要从IO和CPU消耗来考虑成本

  5. 执行器

    作用：根据优化器生成的执行计划，调用存储引擎的接口进行操作，主要过程有:

    • 再次校验用户权限
    • 根据执行计划调用存储引擎
    • 将满足条件的行组成结果集返回给客户端

  6. 存储引擎

    作用：负责数据的存储和提取，位于架构底层。MySQL是插件式的，支持多种存储引擎（如InnoDB, MyISAM, Memory）

5. SQL的执行顺序

更详细的可以参考：
SQL语句执行顺序深度解析 - 《SQL查询实战》 - 严富坤的知识库专栏(yanfukun.com)

select语句通常结构如下：

SELECT [DISTINCT] <select_list>
FROM <table_source>
[JOIN ... ON ...]
[WHERE <where_condition>]
[GROUP BY <group_by_list>]
[HAVING <having_condition>]
[ORDER BY <order_by_list>]
[LIMIT <limit_number>]

其实际的逻辑执行顺序如下：

  1. FROM 确定数据源 联表查询
  2. ON ON筛选
  3. JOIN 连接其他数据表
  4. WHERE 行级过滤
  5. GROUP BY 分组
  6. HAVING 组级过滤
  7. SELECT 选择和计算字段
  8. ORDER BY 排序
  9. LIMIT 数量

更详细的信息如下（这里假设是两张表的联表查询）：

FROM：笛卡尔积

在这一阶段，数据库会对参与查询的两张表执行笛卡尔积（交叉连接），生成虚拟表VT1。（如果有更多的表。通常基于小表驱动大表的规则）

ON：过滤器

对虚拟表VT1应用ON筛选器，筛选出满足ON条件的行保存在虚拟表VT2中。

JOIN：添加外部行

如果指定了参数，如LEFT JOIN， RIGHT JOIN，则数据库还会降未不满足ON条件的行作为外部行添加到VT2中生成新的虚拟表VT3。

  • LEFT OUTER JOIN 把左表记为保留表，对于没有符合条件的右边赋值为NULL
  • RIGHT OUTER JOIN 把右表记为保留表，对于没有符合条件的左表赋值为NULL
  • FULL OUTER JOIN 把左右表都作为保留表

WHERE：行级过滤器

对虚拟表VT3的每一行的数据应用WHERE过滤器，并将满足条件的数据插入虚拟表VT4

GROUP BY：分组

按照GROUP BY的条件将VT4中行唯一值合并成一组生成虚拟表VT5，

Important

如果应用了 GROUP BY，那么后面的所有步骤都只能得到的虚拟表 VT5 的列或者是聚合函数（count、sum、avg等）。原因在于最终的结果集中都要已经是组结果了。

常见的聚合函数包括：

  • COUNT
  • MIN、MAX、AVG、SUM
  • FIRST、LAST

HAVING：组级过滤器

对虚拟表VT5应用组过滤器，会将符合条件的数据保存在虚拟表VT6中

SELECT：

将虚拟表VT6中的结果按照SELECT进行刷选，生成虚拟表VT7
同时，从这一步开始，后面的语句中都可以使用 SELECT 中的别名。

例如以下是一个 查询当天的访客 并根据访问时间倒序排序

SELECT
    visitor_id,
    MAX(visit_time) AS latest_visit_time -- 1. 分组后，取最大时间作为最近访问时间
FROM visit_log
WHERE DATE(visit_time) = '2023-10-27'
GROUP BY visitor_id -- 2. 按访客ID分组，实现“每个访客一条记录”
-- 3. 对分组聚合后的结果按最新时间排序 可以使用别名
ORDER BY latest_visit_time DESC;

DISTINCT

对虚拟表VT7中的结果按照条件进行去重，生成虚拟表VT8

ORDER BY：排序

根据条件对虚拟表VT8进行排序生成游标，注意这里不是虚拟表。因此使用 ORDER BY 子句查询不能应用于表达式。同时，ORDER BY 子句的执行顺序为从左到右排序，是非常消耗资源的。

LIMIT：分页

根据游标从VT8里面选择指定数量行生成虚拟表VT9，并返回调用者

6. Spring中事务的传播行为

  1. REQUIRED

    需要一个事务，如果当前有则加入，没有则新建

  2. SUPPORTS

    支持事务，有则加入，无则无事发生

  3. MANDATORY

    强制当前存在事务，没有则抛出异常

  4. REQUIRES_NEW

    需要一个新的事务，不管当前有没有都会创建一个新的事务

  5. NOT_SUPPORTED

    不支持事务，如果当前有事务则挂机

  6. NEVER

    如果当前存在事务会抛出异常

  7. NESTED

    如果当前有事务，则使用嵌套事务，没有则新建事务

    Tip

    嵌套事务：子事务会创建保存点

    如果子事务执行成功，则会跟随父事务的提交而提交

    如果子事务执行失败，则子事务的回滚状态并不会影响到父事务的回滚状态，但是如果在同一个事务中则会影响

    注意：事务的回滚状态和异常是两个概念，异常的捕获并不会影响回滚状态的变更

    可以参考：[Spring事务大坑：明明捕获了异常，为什么事务还是回滚了？_异常捕获了会不会事务回滚-CSDN博客

7. Mysql的四大事务特性是如何实现的？

Mysql的四大特性：

  • 原子性

    通过Undo log（撤销日志）保证事务的原子性

  • 一致性

    通过Redo Log（重做日志）保障事务的一致性

  • 隔离性

    通过数据库锁的机制，保障事务的隔离性

  • 持久性

    通过Redo Log+Bing Log保证事务的持久性

8. 为什么mysql的可重复读不会产生幻读

通常情况下可重复读会产生幻读，但是mysql的可重复级别却并不会产生幻读问题，mysql是如何实现的？

mysql的读取分为两种

  • 快照读（Snapshot Read）：普通的、不加锁的select语句。如果在事务里面，读取的是事务开始时的最新数据

    解决幻读的方式：基于MVCC读取历史快照

  • 当前读（Current Read）:需要加锁的读操作，目的是为了获取数据的最新版本。如果在事务里面，读取的是仍然是当前时刻的最新数据。

    解决幻读的方式：临键锁

Important

实际上插入/更新/删除属于特殊的当前读，也需要对对应的数据进行加锁

其中

  • update/delete：会对被修改的数据加上排他锁，还可能加上间隙锁
  • insert：对插入行所在的间隙添加插入意向锁的间隙锁，插入成功后，会对新插入的这条记录加上排它锁

9. 说说记录锁、间隙锁和临键锁

Important

**这些高级行锁一定要有索引才能工作。**如果没有索引，InnoDB会几乎退化锁住整张表，这会严重影响对应表的并发性能甚至导致死锁。

  1. 有索引：

    InnoDB通过索引树快速找到满足条件的记录，然后只对这些索引和它们之间的间隙加锁。
    优点是锁定范围小、精度高、并发高

  2. 没有索引：

    由于没有索引可以利用，InnoDB为了确保查询的正确性和隔离性，不得不进行全表扫描。在全表扫描的过程中，它会对所有扫描到的记录加锁。同时还会对所有可能的间隙加锁，这实际上相当于锁住了整张表。

InnoDB默认使用临键锁，如果是唯一索引等值匹配才会使用记录锁。

记录锁（Record Lock）

作用：锁住索引上的某一条具体记录，防止其他事务修改或删除这条被锁定的记录

例如：

-- id是唯一索引
select * from test where id=1 for update;

-- type为非唯一索引
select * from test where age=15 for update;

需要注意的是：

当where条件用的不是非唯一索引，记录锁不仅仅锁住某一条记录，还可能锁住多条记录和间隙

核心原因是因为非唯一索引的“重复”问题。

最终会导致：

  1. 在type索引树上加锁：包含记录锁+临键锁
  2. 回表到主键id上加锁：只有记录锁

另外非唯一索引的等值匹配如果匹配到了 会使用到前后两个区间的临键锁(实际上是一个临键锁+间隙锁) 否则就是所在区间的临键锁

例如：一个age 索引包含值，10，11，13和20

-- 会锁住(13, 20]这个临键锁
select * from test where age=15 for update;

-- 会锁住(10, 11]和(11, 13]两个临键锁
select * from test where age=11 for update;

间隙锁（Gap Lock）

作用：锁住记录之间的“间隙”，或者第一个记录之前/最后一个记录之后的无限空间。

特点：间隙锁之间并不冲突，多个事务可以同时锁住同一个间隙，只要它们之间插入的位置不冲突（例如大家都锁住id为2-10的间隙，但是一个插入id为5，另一个插入id为6的数据）

临键锁（Next-Key Lock）

定义：记录锁+间隙锁的组合。它锁住的是一个左开右闭（包含下一个记录）的区间(A, B]，可重复读通过临键锁解决了当前读的幻读问题。

临键锁的退化

临键锁退化为记录锁
  1. 唯一索引的等值查询：当使用唯一索引进行精确等值匹配查询(如WHERE id = 10)，且查询的记录确实存在时，InnoDB会将临键锁退化为记录锁，只锁定该行记录。
  2. 主键的等值查询：与唯一索引类似，对主键进行等值查询且记录存在时，也会退化为记录锁。
临键锁退化为间隙锁
  1. 唯一索引的等值查询但记录不存在：当使用唯一索引进行等值查询(如WHERE id = 15)，但该记录不存在时，临键锁会退化为间隙锁，锁定该值前后的间隙。
  2. 非唯一索引的等值查询：对非唯一索引进行等值查询时，即使记录存在，InnoDB也会使用多使用一个间隙锁来防止其他事务插入相同的值。
  3. 范围查询的边界条件：在范围查询的边界条件下，可能会退化为间隙锁。 具体看后面样例4

样例一 间隙锁和插入意向锁 

-- 建表语句
CREATE TABLE `test` (
  `id` int NOT NULL,
  PRIMARY KEY (`id`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci;

insert into test(id) values (5), (10), (15)

-- 事务A
begin;
insert into test(id) values (12);

此时有事务B

-- 情况1：会被阻塞
begin;
select * from test where id > 13 for update;


-- 情况2：不会被阻塞
begin;
select * from test where id > 11 for update;

事务A会对(5, 10)施加插入意向锁 并给id=12上记录锁

事务B很明显是要获取临键锁（记录锁+间隙锁 在mysql中为原子操作）

  • 情况1：

    获取间隙锁(10, 15)和(15, +∞)

    获取记录锁15

  • 情况2：

    获取间隙锁(10, 15)和(15, +∞)

    获取记录锁12和15 此时与情况1的记录锁冲突 导致阻塞

这说明间隙锁不会被插入意向锁阻塞的

但是：

-- 事务B
begin;
select * from test where id > 13 for update;

-- 事务A  会被阻塞
begin;
insert into test(id) values (12);

这说明插入意向锁会被间隙锁阻塞

总的来说：插入意向锁并不阻塞其他事务，但是其他事务通常会阻塞插入意向锁

ps: 我觉得也可能使插入意向锁和间隙锁不互斥 只是因为插入后要获取记录锁，然后才和间隙锁互斥了

Important

需要注意的是，mysql可能会上多个间隙锁

例如有一个表有一个主键和一个唯一索引三个非唯一索引

在插入数据的时候，会添加

  • 对主键的记录锁
  • 对唯一索引的记录锁
  • 对三个非唯一索引添加（两边的间隙锁+等值的记录锁）

非唯一索引的间隙锁会添加两边的！！！！！

样例二 非唯一索引的等值匹配 

-- 有唯一索引id和非唯一索引age
CREATE TABLE `test` (
  `id` int NOT NULL,
  `age` int DEFAULT NULL,
  PRIMARY KEY (`id`),
  KEY `idx_age` (`age`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci;
insert into test(id, age) values(5,5), (10,10), (15,15);

情况1：
-- 事务A
begin;
select * from test where age=10 for update;

-- 事务B 阻塞
begin;
insert into test(id, age) values(8, 8);

情况2：
-- 事务A
begin;
select * from test where age=10 for update;

-- 事务B 阻塞
begin;
insert into test(id, age) values(12, 12);


情况3：
-- 事务A
begin;
select * from test where id=10 for update;

-- 事务B 不阻塞
begin;
insert into test(id, age) values(12, 12);

情况4：
-- 事务A
begin;
select * from test where id=10 for update;

-- 事务B 不阻塞
begin;
insert into test(id, age) values(8, 8);


情况5（情况2的变种）：
-- 事务A
begin;
select * from test where age=10 and id=10 for update;

-- 事务B 不阻塞
begin;
insert into test(id, age) values(12, 12);
  • 情况1和情况2说明了数据库在对非唯一索引进行加锁会用到临键锁，事务A实际上获取了：
    • 临键锁：(5,10]
    • 间隙锁：(10,15)
  • 情况2和情况3说明了在面对唯一索引将会直接使用记录锁，事务A实际上获取了
    • 记录锁：10

对于情况5：因为使用了唯一索引id+非唯一索引可以唯一确定一条数据，因此仍然只获取了记录锁而不是间隙锁

这也印证了前面说的：InnoDB默认使用临键锁，如果是唯一索引等值匹配才会使用记录锁。

样例三 多索引下的抉择 

-- 一个唯一索引和两个非唯一索引
CREATE TABLE `test2` (
  `id` int NOT NULL AUTO_INCREMENT,
  `age` int DEFAULT NULL,
  `type` int DEFAULT NULL,
  PRIMARY KEY (`id`),
  KEY `idx_age` (`age`),
  KEY `idx_type` (`type`)
) ENGINE=InnoDB AUTO_INCREMENT=22 DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci;
insert into test2(id, age, type) values(5,5,5), (10,10,10), (15,15,15);

-- 情况1：
-- 事务A
begin;
select * from test2 where type=10 and age=10 for update;

-- 事务B 被阻塞
begin;
insert into test2(age) values(8);


-- 情况2：
-- 事务A
begin;
select * from test2 where type=10 and age=10 for update;

-- 事务B 不阻塞
begin;
insert into test2(type) values(8);

通过explain计划我们可以发现如下：

这说明了事务A使用了索引age，因此只会对age建立两个间隙锁和一个记录锁。而不会使用type的间隙锁和记录锁，因次情况1中事务B会被阻塞，而情况2的事务B不会被阻塞。

样例四 范围查询

对于范围匹配，上锁基础单位为临键锁，数据库会从左到右依次判断右边界是否在条件里面，如果在则上锁，否则继续往右边判断，直到找到不在搜索条件中的右边界，然后加入这个范围的间隙锁

-- 有唯一索引id和非唯一索引age
CREATE TABLE `test` (
  `id` int NOT NULL,
  `age` int DEFAULT NULL,
  PRIMARY KEY (`id`),
  KEY `idx_age` (`age`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci;
insert into test(id, age) values(5,5), (10,10), (15,15);

-- 事务A
begin;
select * from test where id >=5 and id <=10 for update;

-- 下面的语句都会被阻塞
-- insert into test(id) values(2);
-- insert into test(id) values(12);

-- 如果查询条件为[5,10)  则会阻塞(-∞, 10] 因为10不在范围查询条件里面

-- 但是注意唯一索引和非唯一索引的加锁区别

-- 情况一：加锁范围是 (-∞,10)  （会给id上锁）
-- select * from test where id >=5 and id <8 for update;

-- 情况二：加锁范围是 (-∞,10]  （会给age和id都上锁，其中id会回表上锁）
-- select * from test where age >=5 and age <8 for update;
-- 下面都会被阻塞
-- select * from test where age=10 for update;
-- select * from test where id=10 for update;
  1. 第一个临键锁右边界为5，5在条件里面，添加临键锁(-∞,5]
  2. 第二个临键锁右边界为10，也在条件里面，添加临键锁(5,10]
  3. 第三个临键锁右边界为15，不在条件里面。添加间隙锁(10,15)

总结

总的来说，加锁的基础单位是临键锁

mysql会从叶子结点从左到右依次扫描已有的数据，不存在的意思指对应的数据是否在查询条件里面,例如id有5,10,15 查询条件为[5, 10):
那么先扫描到5存在 则加入临键锁(-∞, 5)
然后扫描到10 发现不存在 加入间隙锁(5,10) 结束

  • 唯一索引
    • 等值查询
      • 存在：临键锁退化为记录锁
      • 不存在：临键锁退化为间隙锁 （实际上跟范围查询一样的）
    • 范围查询：临键锁+间隙锁
  • 非唯一索引
    • 等值查询 （实际上跟范围查询一样的）
      • 存在：**临键锁+间隙锁 **
      • 不存在：间隙锁
    • 范围查询：临键锁 （最后一个不会退化为间隙锁）

这样看来，加锁的基础单位为临键锁，加锁的规则基本分为以下三类

  • 唯一索引+等值查询+存在：记录锁
  • 非唯一索引+范围查询：临键锁
  • 其他：临键锁（存在）+间隙锁（不存在）

Important

需要注意的是，如果加锁的时候没有用到索引，那么数据库实际上会锁定所有的记录+间隙，相当于锁表了，平时开发一定要注意规避这类问题。

10. 介绍下MVCC

MVCC（Multi-Version Concurrent Control）：多版本并发控制，是用来解决读写冲突的无锁并发控制，目的是高并发

在没有MVCC的情况下，数据库通常通过锁来实现并发控制，这不仅会使数据库的并发能力变差，大量的锁还会带来严重的性能开销和死锁风险。

特点：

  • 读写不阻塞
    • 读操作访问历史版本，不影响在写的操作
    • 写操作创建新版本，不影响在读的操作
  • 降低死锁概率
  • 解决快照读的幻读问题

MVCC的核心思想是为每一行数据维护多个版本。当一个事务需要读取数据室，数据库会基于某个时间点给他提供一“快照”，这个快照包含了当时所有已提交的数据版本，这样读操作和写操作就会在互不干扰的情况下处理同一份数据的不同版本。

实现原理

  1. InnoDB引擎为了支持MVCC，在每行数据中还添加了一些隐藏字段：

    • DB_ROW_ID（行id）

      隐含的自增ID（隐藏主键）。如果表有主键则用表主键；如果表没有主键，InnoDB会自动生成一个隐藏的主键，并根据其生成聚簇索引。

    • DB_TRX_ID（最近修改事务id）

      记录最后一次对本行数据做修改操作的事务id

    • DB_ROLL_PTR（回滚指针）

      指向该行数据在Undo Log中的上一个历史版本。所有旧版本的数据都通过这个指针连接成一个链表，成为版本链

  2. Undo Log

    这是 MVCC 的基石。当你更新或删除一条记录时，旧版本的数据不会被立刻物理删除，而是会被拷贝到 Undo Log 中。这个旧版本的数据就成为了该行数据的一个历史快照。

  3. Read View(读视图)

    当一个事务执行快照读（普通的 SELECT ... 语句，不包括 SELECT ... FOR UPDATE 或 SELECT ... LOCK IN SHARE MODE 这类加锁读）时，InnoDB 会为它创建一个 Read View。

    在一个事务中假设有多个快照读：

    • 隔离级别为读已提交（RC）

      每个快照读都会生成并获取最新的Read View

    • 隔离级别为可重复读（RR）

      只会在

    Read View决定了这个事务能看到哪些版本的数据。Read View 主要包含以下关键信息：

    • m_ids:创建read_view时（即首次快照读而不是事务开始），当前数据库中所有未提交的活跃事务的事务id集合
    • m_creator_trx_id:读视图所在的事务id 如果版本链的事务id等于这个说明是自己修改的 直接可见
    • m_up_limit_id:活跃事务m_ids中的最小的事务。小于这个ID的数据版本均可见
    • m_low_limit_id:目前出现过的最大的事务ID+1，大于这个事务的id都不可见

    如果在后两者之间则使用二分法查找对应的事务id是否在m_ids里面 如果在说明事务还活跃着没有提交说明不可见，不在则说明可见

  4. Purge机制

    由于 MVCC 需要保留旧版本数据，Undo Log 会不断增长。InnoDB 有后台的 purge 线程，它会定期检查和删除那些不再被任何活跃事务的快照所需要的旧版本数据，从而回收空间。

11. 什么是意向锁

基本概念

意向锁（Intention Lock）是InnoDB实现多粒度锁机制的关键组成部分，它表示“某个事物打算在表的某个部分加锁”，这意味着它属于表级锁，意向锁分为两种：

  • 意向共享锁（IS：Intention Shared Lock）：表示事务打算在表的某些行上设置共享锁
  • 意向排它锁（XS：Intention Exclusive Lock）：表示事务打算在表的某些行上设置排它锁

事务在获取行锁之前，必须先获取表级的意向锁

作用

意向锁的主要作用就是方便快速判断表是否被锁定，避免逐行检查锁状态

意向锁之间并不互斥，他们只是表示表里面有某些行上了对应的锁，

意向锁和表级锁互斥，例如有某个事务想要对表上互斥锁，那么它只需要判断表上是否有意向排它锁即可，不用去逐行检查行数据是否被上了排它锁。

总的来说，意向锁是为表级锁服务的

和插入意向锁的区别

尽管插入意向锁包含意向锁三个字，但插入意向锁实际上是一种特殊的间隙锁，属于行锁的范畴，和表级锁的意向锁并没有关系

12. 索引下推是什么？

索引下推(Index Condition Pushdown, ICP)是Mysql5.6引入的一项重要优化技术，它主要改变存储引擎层和服务器层的写作方式。

什么是下推？

下推的内容主要是索引列的部分判断逻辑，将过滤操作从服务层下推到存储引擎层。例如:

-- 表结构
CREATE TABLE orders (
    id INT PRIMARY KEY,
    user_id INT,
    create_time DATETIME,
    status VARCHAR(20),
    INDEX idx_user_status (user_id, status)
);

-- 查询：找出用户1001状态包含"paid"的订单
SELECT * FROM orders 
WHERE user_id = 1111
AND status LIKE '%paid%';


-- 再比如有联合索引（a,b,c）
-- 对于查询语句select * from t where a=1 and c=1;
-- 尽管只有a用到索引，但是该语句能使用ICP
  • 没有ICP
    1. 存储引擎首先找到所有user_id = 1111的记录（假设有1000条）
    2. 存储引擎将这1000条数据传输给服务层
    3. 服务层检查每条记录是否匹配要求的通配条件
    4. 最终返回符合条件的记录（假设10条）
  • 有ICP
    1. 存储引擎首先找到所有user_id = 1111的记录（假设有1000条）
    2. 存储引擎内部检查是否符合通配条件
    3. 返回符合条件的记录（假设10条）给服务层
    4. 服务层返回数据

索引下推的必要条件

  • 查询必须使用二级索引（聚簇索引不会使用到索引下推）
  • where条件中至少有一个条件涉及索引列但无法用于索引扫描

存储引擎通过索引只能看到索引包含的列值，这就是为什么非索引列条件无法下推的根本原因。

所以ICP通常和联合索引绑定。ICP通常和主键没关系是因为不会通常不会建立包含主键的联合索引

13. 非聚簇索引为什么不存数据地址值而存储主键？

因为聚簇索引中有时会引发分页操作、重排操作数据有可能会移动

14. 说说explain

explain用于查看一个sql语句的执行计划

执行计划主要包含以下信息：

  • id：查询序号
  • select_type：查询类型
  • table：输出结果集的表
  • partitions：匹配的分区
  • type：表的连接类型
    1. system：表中只有一行数据，const的特例。通常是系统表
    2. const：主键或者唯一索引的等值匹配
    3. eq_ref：在进行多表连接（JOIN）查询时，对于来自前一个表的每一行，在当前表中只能找到唯一的一行与之匹配。通常出现在使用主键或唯一索引作为关联条件的连接查询中。
    4. ref：非唯一索引的等值匹配
    5. fulltext:使用全文索引的时候是这个类型。要注意，全文索引的优先级很高，若全文索引和普通索引同时存在时，mysql不管代价，优先选择使用全文索引
    6. ref_or_null:类似于 ref，但 MySQL 必须额外对包含 NULL 值的行进行搜索。
    7. index_merge:表示查询使用了两个以上的索引，最后取交集或者并集，常见and ，or的条件使用了不同的索引，官方排序这个在ref_or_null之后，但是实际上由于要读取多个索引，性能可能大部分时间都不如range
    8. range：索引范围查询
    9. index：索引全表扫描
    10. all：全表扫描 性能最差
  • possible_keys：查询时可能用到的索引
  • key：实际用到的索引
  • key_len：索引字段的长度
  • ref：列与索引的比较
    1. const：表示使用了常量值
    2. 某个列名：表示使用了普通索引进行查找
    3. 某个函数名：例如UPPER（）导致索引失效或者部分失效
    4. NULL：没有使用索引
  • row：预估扫描的行数
  • filtered：表示存储引擎返回的数据在server层过滤后，剩下多少满足查询的记录数量的比例
  • extra：执行情况描述和说明
    1. distinct：在select部分使用了distinct
    2. Using filesort：表示使用了内存/磁盘排序
    3. Using Index：使用了覆盖索引
    4. Using Index Condition：使用了索引下推
    5. …

15. BinLog的三种模式

  1. STATEMENT

    • 记录内容：记录实际执行的sql语句
    • 优点：
      • 日志文件小，IO压力小
      • 主从复制兼容性好
    • 缺点：
      • 某些函数在主从复制不一致，如NOW()、UUID()
    • 适用场景：适合sql简单且不会调用不确定的函数的场景

  2. ROW（5.7.7+后成为默认模式）

    • 记录内容：记录每一行的数据变化
    • 优点：
      • 数据一致性高，不存在不一致的情况
    • 缺点：
      • 日志文件大
      • 主从复制需要更多资源
    • 适用场景：要求数据一致性的场景，5.7.7之后成为mysql的默认模式

  3. MIXED

    记录内容：默认使用statement模式，特殊情况用row模式

    切换条件：包含不确定函数时

16. 说说三大日志的刷盘时机

redo log

redo log的刷盘由innodb_flush_log_at_trx_commit参数控制：

  • 0：每秒刷盘一次，事务提交时不主动刷盘（可能丢失最多1秒的数据）
  • 1：每次事务提交时都刷盘（默认，最安全）
  • 2：每次事务提交时将redo log写入操作系统缓存，但不刷盘（MySQL宕机不会丢失数据，但操作系统崩溃会丢失）

此外，还有以下情况会触发redo log刷盘：

  • redo log buffer空间不足时
  • 后台线程定期刷盘
  • MySQL正常关闭时
  • checkpoint时

bin log

binlog的刷盘由sync_binlog参数控制：

  • 0：依赖操作系统刷盘（性能最好，安全性最差）
  • 1：每次事务提交时都刷盘（默认，最安全）
  • N：每N次事务提交时刷盘一次

undo log

undo log的刷盘时机：

  • undo log是作为redo log的一部分记录的，所以其持久化依赖于redo log的刷盘
  • undo页的修改也会产生redo log，因此redo log刷盘时也保证了undo log的持久化
  • 事务提交时不会立即删除undo log，因为可能需要用于MVCC

17. InnoDB是如何实现事务的？以update为例

假设待执行sql为: UPDATE t SET col1='new' WHERE id=1

  1. 事务开始：分配事务id

  2. 数据查询：

    • 根据聚簇索引定位到id=1的记录的所在页，检查所在页是否在Buffer Pool里面，如果不在则从磁盘加载
    • 获取记录的行锁（如果有必要）

  3. 修改数据

    • 记录Undo Log日志，将旧值写入undo log
    • 在Buffer Pool根据sql更新数据页
    • 记录redo log日志，将物理页变更到redo log buffer中

  4. 事务提交（2PC：Two-Phase Commit）

    • Prepare阶段
      • 在redo log中标记事务状态为prepare
      • 将redo log日志刷盘
    • Commit阶段
      • 将binlog刷盘
      • 在redo log中标记事务状态为commit

  5. 释放资源

    主要包括锁消除、更新事务状态为已提交等动作

  6. 后台处理

    • 脏页刷盘
    • purge清除不需要的undo log和不用的历史版本链数据

18. Mysql三大日志

binlog是Mysql服务层日志

redolog和undolog则独属于InnoDB,属于引擎层日志

BinLog

  • 作用：主要记录数据库变更操作，Mysql使用任何引擎都会有BinLog日志

  • 日志类型：逻辑日志

  • 用途：主要用于主从复制+数据恢复，保障事务的持久性

Redolog

WAL(Write-Ahead Logging):在数据本身的变更写入磁盘之前，必须先将描述这个变更的日志（redo Log）写入磁盘。也就是日志先行。

具体步骤如下：

  • 修改数据
  • 生成日志
  • 日志刷盘
  • 数据刷盘

好处：

  • 保证持久性，实现崩溃恢复
  • 将随机写（写数据是随机写）变为顺序写（写redolog是顺序写），极大提升性能。通过WAL，数据库只需要在提交时完成一次快速的顺序IO（日志刷盘）就可以返回。而耗时的随机IO（脏数据刷盘）可以放到后台批量、缓慢的进行
  • 组提交，由于redlog是顺序追加，所以可以将多个并发事务的redolog刷盘操作合并成一次刷盘，大大降低了磁盘IO的次数。
  • 作用：记录数据页的物理变化
  • 日志类型：物理日志；顺序IO
  • 用途：崩溃恢复时结合checkpoint和binglog进行数据恢复，保障事务的一致性和持久性
  • 好处：
    • 降低刷盘频率
    • 顺序写入

Undolog

  • 作用：记录修改前的行数据快照
  • 日志类型：逻辑日志
  • 用途：MVCC、保障事务的原子性（失败撤销）

Important

除了上面三个日志

还有：

  • 一般查询日志
  • 慢查询日志
  • 错误日志

19. Change Buffer

Change Buffer 是 Buffer Pool 中的一块特殊内存空间，其主要作用是缓存对非唯一二级索引（non-unique secondary index）页的更改操作。

  • 目的：将针对二级索引的随机写 I/O 合并起来，变成顺序写 I/O，从而极大地提升写操作的性能。
  • 前提：操作只针对非唯一的二级索引。对主键索引和唯一索引的修改无法使用 Change Buffer。

假设我们有一张表 users，在 age 字段上有一个非唯一的二级索引。执行 INSERT INTO users (name, age) VALUES ('Alice', 30);：

  1. 判断：InnoDB 发现要修改 age 索引，且该索引页不在 Buffer Pool 中。

  2. 缓存操作：它不会立即去磁盘读取 age=30 对应的索引页，而是将 “在 age 索引中为 30 这个值添加一条记录” 这个操作信息缓存到 Change Buffer 中。

  3. 标记事务完成：事务可以很快完成，因为只在内存中记录了一个操作，避免了等待磁盘 I/O。

  4. 后续合并（Merge）：

    • 当未来某个时刻（例如，有 SELECT 查询需要读取 age=30 的索引页时），InnoDB 才会将对应的索引页从磁盘加载到 Buffer Pool。

    • 此时，Change Buffer 会将其缓存的关于这个页的所有更改操作（如插入、删除记录）应用到新加载的索引页上，使其保持最新状态。

    • 之后，这个被更新过的索引页就变成了 Buffer Pool 中的脏页，等待后台线程刷盘。

最后change buffer的变动也会跟随redo log日志保存。

20. 说说CheckPoint

LSN(Log Sequence Number, 日志序列号)：代表redolog的总写入量。LSN 是数据库内部的一致性坐标。通过比较数据页的 pageLSN 和 redo log 的 LSN，InnoDB 可以精确知道哪些修改已经落盘，哪些还需要重做。

崩溃恢复

数据库的崩溃恢复大致会经历以下过程：

  • 数据库重启：InnoDB读取redolog文件头中记录的最后一个CheckPoint的LSN
  • 重放redo log：从上面检查点的位置开始，扫描并重放后续的redolog记录
  • 快速完成：检查点之前的数据肯定已经完成了磁盘持久化，这样就避免了从头开始重放redolog

CheckPoint定义

CheckPoint类似游戏“存档点”，其核心作用是：

  • 将buffer pool中的脏数据刷盘到磁盘中
  • 推进并标记redo log的复用点，标识到此的所有的数据都已经完成持久化

必要性

如果没有CheckPoint，那么数据库会面临以下几个问题：

  • redo log无限增长
  • 崩溃恢复时间过长

21. Mysql主从复制原理与方式

原理

Mysql主从复制指将一个Mysql数据库（主库）的数据复制到一个/多个Mysql数据库的过程。其核心思想是主库记录所有数据变更，从库读取并应用这些数据变更。

核心步骤：

  1. 日志记录：主库将所有数据变更事件记录在二进制日志（bin log）中
  2. 日志传输：从库的IO线程连接到主库，请求获取binlog内容主库的binlog dump线程将日志内容发送给从库
  3. 日志重放：从库的sql线程读取中继日志（relay log）中的事件，重新执行这些sql语句，实现数据同步

工作流程

  1. 主库记录数据变更
    • 事务提交时，主库将变更写入binlog
    • 使用sync_binlong参数控制刷盘策略
  2. 从库请求日志
    • 从库IO线程使用主库账号连接主库
    • 请求从指定位置开始获取binlog
  3. 主库发送日志
    • 主库binlog dump线程读取binlog日志
    • 通过TCP连接发送给从库IO线程
  4. 从库存储日志
    • 从库IO线程将收到的binlog日志写入relay log
    • 记录已接收到的binlog位置信息
  5. 从库应用日志
    • 从库sql线程读取relay log中的事件并解析成sql
    • 在从库执行sql语句
    • 记录已执行的relay log位置信息

同步方式

  1. 异步复制（默认）

    Mysql主从复制默认是异步的。在异步复制中，主库写入binlog日志后即可成功返回客户端，无须等待binlog日志传递给从库的过程。

  2. 同步复制

    Mysql的主节点需要等待所有的从节点返回数据复制成功的信息，才可以提交事务

  3. 半同步复制

    Mysql的主节点仅需等待一个或一部分从节点返回数据复制成功的信息，便可以提交事务

22. Mysql主从同步延时产生原因

延时主要发生在主从复制的两个阶段：传输阶段和重放阶段

  1. 硬件问题
    • CPU资源不足
      • 主库：高并发写入
      • 从库：在旧版本中，从库的sql线程是单线程导致cpu利用率不足
    • 磁盘IO性能差
      • 主库：主库使用了每次事务提交都对binlog刷盘，产生大量随机IO。
      • 从库：
    • 网络宽带不足或延迟高
      • 大量业务请求打满宽带
      • 跨机房同步导致延迟较高
    • 主从配置部队称
  2. 业务问题:
    • 长事务
    • 从库中的慢查询争抢资源

3. count(*)、count(1)和count(列名)的区别

  1. count(*):

    统计所有行数，包括NULL值，推荐写法

  2. count(1):

    统计所有行数，相当于每行返回一个常量值1然后计数，和count(*)性能几乎相同

  3. count(列名)：

    统计指定列有数据的的数量，忽略NULL值

4. 说说exists和in的区别 

-- IN 
SELECT * FROM A WHERE id IN (SELECT id FROM B);
-- EXISTS
SELECT * FROM A WHERE EXISTS (SELECT 1 FROM B WHERE B.id = A.id);

使用In的时候，sql会先查询B表再查询A表；exists则是先查询A表，再检查B表。我们要遵循的基本规则是小表驱动大表，也就是先查询小表再查询大表，如果A表数据较少，则用in比较好，反之A表数据比较多，则适合exits

需要注意的是：exists的子查询实际上并不返回任何数据，因为它只作检查，因此仅返回True或者False

5. Distinct和Grout By

结论：Mysql8.0之前，group by会对结果进行隐式排序，因此dintinct的效率略微高于group by；但是在Mysql 8.0之后，Mysql删除了隐式排序，此时两者效率近乎等价

但是通常更推荐group by:

  • 语义更为清洗
  • group by可以对数据进行更为复杂的处理（例如对其他自带的聚合）
  • group by之后还可以对数据进行过滤操作

6. 索引失效的场景

  1. 不符合最左匹配原则，例如联合索引或者like左模糊匹配
  2. 索引列（参与运算 or 隐式转换 or 函数计算）
  3. or关键字使用不当：例如其中一个没有索引导致全表
  4. 普通索引使用不等于
  5. 普通索引使用is not null
  6. 普通索引使用not in
  7. 查询数据太多回表代价太大，放弃索引使用全表扫描

1.MySQL 数据库作发布系统的存储，一天五万条以上的增量，预计运维三年,怎么优化？

Tip

日增量5w条，那么三年预计数据大概在5500w左右
所有的系统架构设计 都要从未来大致的数据规模出发

  • 数据库层面：

    1. 合适的存储引擎，毫无疑问选择InnoDB，支持事务、行级锁以及崩溃恢复能力，并且支持高并发读写场景
    2. 合理的字段设计，满足最小需求原则
    3. 合理的表结构设计，尽量不要设计大表，如果有则最好拆分小表
    4. 表结构按照业务需求可以进行一定程度的冗余，避免频繁的联表查询
    5. 合理的索引设计

  • 业务层面：

    1. sql编写尽量使用索引，只查询出需要使用的字段，同时尽量使用覆盖索引，避免回表操作。
    2. 大查询拆分，例如联表操作可以简化两个查询操作
    3. 批处理操作

  • 架构层面：

    1. 数据读写分离

    2. 数据分区

    3. 数据归档

    4. 垂直分库/水平分表

      分库分表常用思路：

      • hash取模：通常对用户id

        优点是平均；缺点是扩容较为麻烦

      • range范围：通常对用户id 或者 时间

        优点是扩容简单；缺点是流量存在热点（最新的数据访问可能更频繁）

      • hash和range结合

      • 一致性hash

  • 监控：开启关键指标监控，如连接数、慢查询数量等

2. 大批量（如千万级）数据导入该怎么优化？

  1. 导入过程中删除不重要的索引，在导入完成后再重新建立

  2. 使用批处理，通常每个批次的数量设置为1000，java层面的批处理基于preparedStatement。（通常一个语句不要超过1M大小）

    PreparedStatement是Java JDBC API中的一个重要接口，用于执行预编译的SQL语句。它比普通的Statement更高效、更安全，特别是在需要多次执行相似SQL语句时。

    主要特点：

    • 预编译：sql语句会在首次执行时被预编译并缓存，后续执行只需使用编译好的语句和数据填充
    • 防止sql注入：自动处理参数转义，有效防止SQL注入攻击
    • 类型安全：支持参数的类型检查

  3. rewriteBatchedStatement设置为true，这是JDBC层面的一个重要优化，作用是改写多条相同的插入sql为一条。通常和批处理一起。

  4. 合适的事务管理，每个事务数据不能太多也不能太少 可以10个批次1w条一个事务

  5. 开启多线程，并发导入

  6. 如果业务允许，可以预填充id

  7. 尽量保证数据的有序（针对主键）插入，这样会降低主键索引的插入成本

3. 分库分表会带来什么问题，如果解决？

是否分库分表要根据业务规模来决定。分库分表会带来以下问题：

  1. 分布式事务问题
    根据业务需求使用合适的分布式事务中间件（有最终一致性和强一致性）

  2. 跨节点关联查询

    解决方法：

    • 建立全局表

    • 字段冗余

    • 数据组装

  3. 跨节点统计、排序

  4. 全局主键
    使用雪花id

  5. 数据迁移

    要分两个阶段进行数据迁移，先进行数据双写，然后进行历史数据同步，最后比较，测试。

4. 一条sql执行时间过长如何优化？

  • sql层面
    1. 首先根据提取慢sql，使用explain检查该语句的执行计划，检查索引是否生效且合理，或者多表连接索引是否生效
    2. 检查查询字段，是否可以简化，从而使用索引覆盖或者降低排序等内存消耗
    3. 是否遵循小表驱动大表
    4. 是否由于深分页导致慢sql
    5. 大查询拆解成小查询，提高系统吞吐量
  • 数据层面
    1. 大表是否需要拆分，包含分库分表等策略
    2. 是否由于数据库压力导致吞吐量下降，可以采用读写分离
    3. 是否需要冗余字段，降低联表压力
    4. 是否允许一定时间内的旧值存在，可以采用缓存+定时任务查询自动更新
