基础

1. 为什么要使用MQ？有什么优缺点

主要是因为MQ的优点，或者说是使用MQ的核心目的主要是：

  • 任务异步

    场景：部分比较耗时的操作，例如报表导出、发短信、日志存储等操作

    MQ作用：只需将任务添加到消息队列后即可返回给用户，后续耗时操作交给消息队列负责处理，提高了系统的即时响应速度

  • 业务解耦

    场景：上游系统调用若干下游系统，

    问题：上下游之间耦合严重（任意一个下游系统错误都会导致整个业务出错；每次添加一个下游系统时，上游系统都要修改代码）

    MQ作用：上游系统只需将消息丢给消息队列，下游系统只需要按需订阅自己需要的消息进行消费，降低了系统的耦合度

  • 流量削峰

    场景：上游系统突发流量，例如大量报表导出任务

    问题：下游系统很可能扛不住突发的大流量，需要对流量进行一定限制

    MQ作用：将任务请求放进消息队列中排序，下游系统按照自己消费能力消费消息，提高系统的可用性和稳定性

除此之外，部分消息队列还提供以下特性：

  • 顺序性
  • 持久化
  • 最终一致性

MQ的缺点

没有银弹，引入MQ也带来了一些复杂性和挑战：

  • 系统的复杂性提高
  • 系统可用性降低：消息队列成为了最脆弱的地方，如果消息队列崩溃，那么相关业务也会出错
  • 消息的丢失
  • 消息的重复性消费
  • 消息的持久化
  • 消息的一致性

2. MQ有哪些常见问题？怎么解决？

  1. 消息的顺序问题

    在Rocketmq里面，Message Queue是典型的FIFO，天然顺序。因此我们只需要确保顺序生产顺序消费即可。
    具体来讲就是使用一个线程生产消息，使用一个queue存储消息，使用一个线程消费消息，这样就严格保证了消息的顺序消费。

    Important

    这种方案很明显严重限制了应用的吞吐量。

    也因此，我们更应该结合业务的角度去确保消息的顺序性，而不是单单依靠消息系统来确保。

    例如将一个用户的消息（下单、发货、收获、结算）都发送到一个消息队列中，同时确保这些消息只会被一个消费者消费即可，那么就是总体无序（还有很多其他的用户消息），业务有序。

  2. 消息重复消息

    由于网络问题导致重复投递；或者消费者消费后宕机，重启后再次受到消息等多种原因，消费者可能会面临消息重复消费的问题，

    解决思想：保证业务的幂等性

    解决方案：

    • 基于数据库

      • 对于新增数据：使用数据库唯一索引确保消息的幂等性。在封装到消息队列中时添加全局唯一id，并在新增到数据库时将这个id保存起来作为唯一键，后续如果重复消息就会插入失败，避免了消息的重复消费

      • 对于修改数据：通常需要根据业务字段确保。比较常规的解决方法是采用乐观锁的方案，例如记录上一次修改的原始值，状态值等信息，在更新数据的时候进行期望值的匹配，如果匹配成功则修改成功，否则修改失败，

    • 基于redis

      • 同样我们需要给消息一个全局唯一的id，在消费的时候我们通过redis获取一个分布式锁，通过分布式锁来限制消息的重复性消费

        在实际业务中，我们可以设置一定的过期时间，例如3600s，超过这个时间我们就认为消息不会重复性消费了

        同时我们可以采用redis+数据库两层来判断

        先判断redis，再用数据库兜底

3. Rocketmq的组成和常见概念

RocketMq总体由四个部分（角色）构成：

  • NameServer：轻量级无状态的路由管理中心

    核心职责：

    • 服务注册：所有的broker都需要向NameServer注册自己的信息
    • 心跳维护：Broker定期向NameServer发送心跳，证明自己还活着
    • 路由发现：生产者和消费者客户端会主动从NameServer拉取最新的路由信息，知道哪个Topic的哪个Queue在哪个Broker上

  • Broker：消息的存储和转发中心

    • Broker 内部维护着一个个 Consumer Queue，用来存储消息的索引，真正存储消息的地方是 CommitLog（日志文件）
    • 单个 Broker 与所有的 Nameserver 保持着长连接和心跳，并会定时将 Topic 信息同步到 NameServer，和 NameServer 的通信底层是通过 Netty 实现的。

  • Producer：消息的生产者

    RocketMq提供了三种方式发送消息：

    • 同步发送：生产者将消息发送给broker后，会一直等待broker返回的结果
    • 异步发送：生产者将消息发送给broker后，不用等待broker的结果，但是需要提供一个回调函数，broker处理完消息后，会异步地调用这个回调函数生产者的发送结果
    • 单向发送：生产者将消息发送给broker后，不用等待broker的结果，也不会提供回调函数

  • Consumer：消息的消费者

    支持PULL（主动拉取消息消费）和PULL（被动消费）两种消费模式，支持集群消费和广播消费

还有一些基础概念

  • Message

    具体的消息，每条消息必须具备一个主题和一个可选的tag

  • Topic

    消息的归类，相当于对消息的一级分类，是必选属性

  • Tag

    可以简单看作消息的二级分类，是可选属性

  • Group

    通常指Consumer Group,这个概念是针对消费者体现的，多个消费者可以归属到一个消费组里面

  • Message Queue

    消息队列，一个Topic可以设置多个消息队列，也就是一个topic对应多个Queue

  • Offset

    在 Topic 的消费过程中，由于消息需要被不同的组进行多次消费，所以消费完的消息并不会立即被删除，这就需要 RocketMQ 为每个消费组在每个队列上维护一个消费位置（Consumer Offset），这个位置之前的消息都被消费过，之后的消息都没有被消费过，每成功消费一条消息，消费位置就加一。

5. RocketMq的消费模式

  1. 集群消费

    如果消息投递到一个消费组里面，那么仅会有一个消费者能够消费该消息

    如果消息投递到多个消费组里面，那么每个消费组里面仅会有一个消费者能够消费该消息

  2. 广播消费

    如果一个消息投递到一个消费组里面，那么里面的所有消费者都能够消费到这条消息

6. RocketMq如何保证消息不丢失

消息丢失在三个阶段都可能发生：生产阶段、存储阶段、消费阶段，因此这三个阶段我们都需要考虑

生产阶段

生产者发送消息的方式应选择同步发送或者异步发送来确保消息正确投递到broker

  • 同步发送：如果消息投递失败，应重试直到消息投递成功
  • 异步发送：在回调函数里面检查发送结果，如果发送失败，应尝试重新发送

存储阶段

存储阶段应确保消息能够持久化到磁盘中，因此最好修改刷盘策略为同步刷盘，确保数据能够持久化。但是异步刷盘其实也是可以的

消费阶段

确保消息被正确消费，所以应完全消费之后再手动进行消息的ack确认

8. 如何处理消息堆积？

消息堆积需要从多个维度去进行考虑，分别是生产者和消费者角度

消费者角度

通常我们先考虑消费者情况，通常大类原因包括：

  • 消费能力不足
    • 消费者扩容：前提是当前Message Queue的数量 > 消费者数量，因为一个queue只能绑定一个消费者
    • 消息重建：当消费者数量已经达到了Message Queue数量的时候，这个时候再新增消费者是不能提高消费能力的。这个时候应新建一个临时的Topic，然后将堆积数据消费到临时的Topic中。（相当于用两个Topic来消费业务，提高消息处理能力。也可以只靠临时Topic具备更大的Queue数量然后配对应的消费者）
  • 硬件故障：
    • 消费者宕机
    • GC停顿
  • 消费逻辑异常
    • 代码异常，如死锁，死循环
    • 下游系统调用缓慢超时
    • 消费逻辑资源占用较高，如计算缓慢，IO缓慢

生产者角度

关注较少，如果是非核心业务通常进行业务降级，减少消息的生产

9. Rocketmq如何实现分布式事务？

Rocketmq实现分布式事务是其非常核心且强大的功能之一，它采用的经典的最终一致性方案，称为“事务消息”。这是通过“两阶段提交”和“事务状态回查”组合实现的。它主要包括两个核心阶段：

  • 发送半消息并执行本地事务

    1. 发送半消息
      • 生产者向broker发送一条半消息
      • 半消息对消费者不可见，它被存储在RocketMq的一个特殊topic中
    2. 执行本地事务
      • 半消息发送成功后，生产者开始执行本地事务逻辑
    3. 提交或回滚
      • 本地事务完成后，生产者会根据结果向broker发送提交或回滚的命令
      • 如果是提交命令：将半消息从特殊主题转移到真正的目标主题中，此时半消息变为真正的消息
      • 如果是回滚命令：broker收到命令后，会直接丢地该半消息

  • 事务状态回查

    为了防止网络抖动或生产者宕机导致命令不可达的兜底机制

    • 场景：如果生产者执行完本地事务后，由于网络故障、宕机等原因，一直没有向 Broker 发送 Commit 或 Rollback 指令，这条半消息就会一直处于“悬而未决”的状态。
    • 回查：对于上述这种状态的半消息，需要生产者提供一个回查机制用来判断当前消息关联的本地事务结果是提交了还是回滚了，broker根据此决定是保留消息还是丢地消息

可以看出来，RocketMQ 的分布式事务方案本质上是将分布式事务问题转化为一个本地事务问题+一个消息可靠性投递问题。

优点

  • 最终一致性
  • 高可用性

缺点：

  • 最终一致性
  • 消费者需要幂等
  • 需要实现额外的回查逻辑

10. broker是怎么保存消息的？

一、核心设计思想

  1. 顺序写入

    无论消息来自多少个 Topic，所有新消息都追加写入（Append） 到最后一个文件末尾。磁盘的顺序写速度极快，几乎可以达到磁盘的物理极限。

  2. 内存映射文件（mmap - Memory Mapped File）

    将磁盘文件映射到进程的内存地址空间，对文件的读写操作转化为对内存的操作，避免了传统的 read/write 系统调用，极大提升了性能。

  3. 异步刷盘

    刷盘（将内存数据持久化到磁盘）和主从复制都是异步进行的，生产者发送消息后，只要写入内存就返回成功，从而极快地响应生产者。

二、 存储结构和文件组织

Broker的消息存储主要是三个核心部分组成：

  1. CommitLog（真正的消息仓库）

    所有主题（Topic）的所有消息都顺序地写入同一个文件集合——CommitLog。这是真正的消息实体存储地

    • 文件结构：CommitLog 由多个文件组成，每个文件固定大小（默认1G），写满一个再创建下一个
    • 目的：保证绝对的顺序写入，这是RocketMq高吞吐量的核心，它牺牲了读的便利性，换来写的极致性能

  2. ConsumeQueue（消息的逻辑队列索引）

    为每个 Topic 的每个 Queue 构建的一个消费队列索引文件——ConsumeQueue

    • 文件结构：其中保存的每个消息条目包含以下信息，每个条目固定20字节
      • CommitLog Offset (8 bytes):在CommitLog中的物理偏移量
      • Size (4 bytes):消息长度
      • Message Tag HashCode (8 bytes)：消息Tag的哈希值，用于消息过滤
    • 工作方式：每当消息写入 CommitLog 后，就会有个专门的线程去读取新写入的commitLog数据，然后解析出该消息属于哪个Topic的哪个Queue，然后将对应的索引信息追加到ConsumeQueue的末尾
    • 目的：将物理存储和逻辑队列分离。消费者拉取消息是，需要先查询ConsumeQueue得到索引，再根据索引去CommitLog精准读取消息，有点类似数据库查询了,相当于用空间换取读性能。

  3. IndexFile（消息的哈希索引）

    为了支持按 Message Key 或 时间范围 查询消息，RocketMQ 还提供了索引文件（IndexFile）。

    • 工作方式：它通过哈希表的结构来存储索引，key 可以是消息的 Key 或 Unique ID，value 则是该消息在 CommitLog 中的偏移量。
    • 设计目的：主要用于运维命令查询和消息追踪，普通的消息消费流程不依赖它。

什么是页缓存？

为了提升对磁盘文件的读写效率，linux内核会以页为单位，将文件划分为多个数据块。当用户需要对文件中的某个数据块进行读写的时候，linux首先会分配一个内存页绑定对应数据块，将堆磁盘的读写转为对内存的读写。

零拷贝

什么是零拷贝

要理解什么是零拷贝，我们需要先了解传统的“非零拷贝”是如何工作的。

如果服务器要把本机数据的内容发送给客户端，传统做法通常靠read+write

  1. read系统调用：用户线程发起系统调用，要求读取磁盘数据，从用户态->内核态
  2. DMA拷贝：磁盘控制器通过DMA将数据从磁盘拷贝到内核缓存中
  3. CPU拷贝：CPU将数据从内核缓冲区拷贝到用户缓冲区中
  4. read系统调用返回：read系统调用完成，从内核态->用户态
  5. write系统调用：用户线程发起系统调用，准备将数据发送到网络，从用户态->内核态
  6. CPU拷贝：CPU将数据从用户缓冲区拷贝到Socket缓冲区中
  7. DMA拷贝：网卡控制器将数据用Socket缓冲区拷贝到网卡，最后发送出去
  8. write系统调用返回：write系统调用返回，从内核态->用户态

RocketMQ的零拷贝是如何实现的？

RocketMq主要使用了两种零拷贝技术

  1. mmap+write（mmap取代了read）

    这种方式主要用在Broker快速读取commitLog/cosumeQueue文件以相应消费者的拉取请求

    • mmap：内存映射文件，将处于内核态的磁盘数据直接映射到用户进程的虚拟空间中，避免了一次CPU拷贝。相当于用户将对应数据所在的内核缓存的地址拷贝到socket缓冲区中，而无需拷贝数据本身。
    • write：当需要发送数据，broker调用write系统调用

    效果：减少了一次从内核缓存到用户缓冲区的CPU拷贝，用户调用调用write仍需要CPU将数据从内核缓存拷贝到socket缓冲区中

  2. sendFile（真正的零拷贝，取代了read+write）

    这是Linux2.4+内核提供的系统调用，是实现零拷贝的终极武器，RocketMQ在传输消息内容时主要使用此方式

    作用：直接在内核空间中将数据从一个文件描述符传输到另一个文件描述符，完全绕过用户空间

    使用sendFile的流程：

    • Broker 进程调用 sendfile(out_fd, in_fd, ...)，其中 in_fd 是文件描述符，out_fd 是网络套接字描述符
    • DMA 拷贝：DMA 引擎将数据从磁盘拷贝到内核缓冲区（PageCache）。
    • CPU 拷贝：CPU 仅将数据的地址和长度描述符拷贝到 Socket 缓冲区。
    • DMA 拷贝：DMA 引擎依据 Socket 缓冲区中的描述符，直接从 PageCache 中将数据拷贝到网卡。

    整个过程数据完全不需要经过用户空间，只发生了2次上下文切换（sendFile调用和返回）+2次DMA拷贝，做到了真正的零（CPU）拷贝

12. RocketMq的刷盘方式

Rocket提供了两种刷盘策略

  • 异步刷盘（默认策略）

    默认的刷盘策略，为了保证高性能。

    工作流程：

    1. 写入PageCache：生产者发送消息刀刀Broker后，直接被追加到PageCache中。
    2. 立即返回成功：只要消息写入PageCache，就立刻向生产者返回写入成功的消息，此时数据仅在内存中，还没刷入磁盘
    3. 异步持久化：后台存在一个专门的线程，定时将PageCache中的数据异步刷新到磁盘中

    优点：速度快，延迟低

    缺点：存在数据丢失的风险，如果broker突然宕机，还没刷新到磁盘中的数据可能会丢失

    适用场景：追求高吞吐、低延迟的场景，且能够容忍在极端情况下丢失少量消息。例如，业务日志、监控数据、非核心的订单状态同步等。

  • 同步刷盘

    相较于异步刷盘，会等到PageCache中的数据刷新到磁盘后才会给生产者返回发送成功的消息

    优点：数据可靠性高

    缺点：性能较低、延迟较高（通常比异步刷盘慢10倍），因为需要等待磁盘IO操作完成

    适用场景：对数据可靠性要求极高的金融、交易等核心业务场景。例如，支付成功、核心订单创建等消息。

综合

1. 为什么选择RocketMq?

  1. 生态良好：隶属于SpringcloudAlibaba生态，国内社区更为活跃，文档丰富
  2. 学习成本较低：使用java编写，更容易学习和排查问题
  3. 丰富的消息类型和功能：
    • 顺序消费
    • 延时消息
    • 消息过滤：支持基于 SQL92 语法的消息过滤和更轻量的 Tag 过滤
    • 消息回溯
  4. 自带控制台

你为什么选择Rocketmq而不是Kafka？

主要是基于开发效率、功能匹配度和团队技术栈的综合考虑。虽然 Kafka 在吞吐量上可能有理论优势，但 RocketMQ 提供的开箱即用的功能让我能更快速地实现业务需求。

  • 延时任务（核心）：任务创建后延时启动
  • 技术栈更熟悉，社区支持好
  • 后续项目会想SpringCloudAlibaba进化，组件兼容性更高

我的项目不需要 Kafka 极限的吞吐性能（评论采集通常不是海量日志那种级别），但非常看重开发的便捷性、功能的丰富性和实现的简洁性。RocketMQ 提供的延时消息、消息过滤等功能都是开箱即用的，让我能专注于业务逻辑而不是基础组件的能力建设，这是我当时选择它的最核心原因。
