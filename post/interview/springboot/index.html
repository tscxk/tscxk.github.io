基础

1. 对象的创建过程

参考：JVM系列(六) -对象的创建过程_jvm创建对象的过程-CSDN博客

对象的创建过程通常包含：

  1. 类加载检查

    这一步实际上就是类装载的过程，毕竟创建对象前要先检查类的存在

    （因为类加载默认是懒加载，只有用到的时候才需要加载，所有通常需要类加载检查）

  2. 分配内存空间

    主要有两种方式：

    • 指针碰撞法（内存规整，当垃圾回收器采用Serial和ParNew这种基于标记-整理算法的，通常使用这种分配方式）
    • 空闲链表法（内存不规整，当垃圾回收器采用CMS这种基于标记-清除算法的，通常使用这种分配方式）

    多线程内存分配冲突主要采用两种策略：

    • CAS
    • 本地线程缓冲技术（先预分配一部分给本地线程，之后本地没了再采用CAS获取）

  3. 初始化零值

    例如给int赋值为0，普通对象赋值为null

  4. 设置对象头

    例如hashCode、年龄、数组长度大小等属性设置

  5. 初始化

    例如给对象的成员变量设置用户的指定的初始值，执行构造方法等

2. Bean的生命周期

参考：面试突击80：说一下 Spring 中 Bean 的生命周期？-阿里云开发者社区

Spring中Bean的生命周期从创建到销毁主要有5个过程：

  1. 实例化：分配内存空间+设置默认值
  2. 属性填充：进行依赖注入，如属性注入和set注入
  3. 初始化：
    • 执行各种aware通知方法
    • 执行前置方法
    • 执行初始化
    • 执行后置方法
  4. 使用：在程序中使用Bean对象
  5. 销毁：将Bean对象进行销毁操作

3. SpringBoot是怎么解决循环依赖的？

SpringBoot是通过三级缓存机制来解决单例作用域的Bean之间的属性注入或set注入类型的循环依赖。

需要注意的是，Spring无法解决构造器注入导致的循环依赖，会直接抛出BeanCurrentlyInCreationException 异常，原因是因为：对于原型不会缓存，对于构造器注入无法提前暴露对象的引用

核心机制：三级缓存

Spring内部实际上维护了三个Map容器，即我们常说的三级缓存

  1. 一级缓存(Singleton Objects)

    • 存放内容：完全初始好的，成熟的Bean

    • 作用：业务代码直接使用的Bean

  2. 二级缓存(Early Singleton Objects)

    • 存放内容：在循环依赖中临时存储 提前暴露的 Bean 引用（可能是代理对象也可能时原始对象）

    • 作用：防止重复创建

  3. 三级缓存（Singleton Factories）

    存放内容：Bean的对象工厂

    作用：AOP代理的关键。当Bean对象存在AOP时，返回代理对象。当Bean对象不存在AOP时，返回原始对象

为什么需要三级缓存？
按照我自己的理解，是因为确保在产生循环依赖的问题时，二级缓存中的代理对象和最终放入一级缓存的对象是同一个，否则不能判断是否要重新生成代理对象

5. Springboot自动装配机制

一文彻底弄懂spring boot自动装配的过程 - lgx211 - 博客园

SpringBoot自动装配的核心机制：

  1. 收集自动配置类
  2. 条件检查
  3. 注入所需要的Bean
  4. 允许用户覆盖自动配置

具体的，主要通过@SpringBootApplication注解，它由三部分组成：

  • @SpringBootConfiguration：标记为一个配置类

  • @componentScan：扫描所有spring组件

  • @EnableAutoConfiguration:

    包含@AutoConfigurationImportSelector注解，自动配置类的核心机制

    该注解会从配置类文件读取所有的配置类文件，并将它们导入到应用上下文中。

    随后每个自动配置类通常都会使用 @Conditional 系列注解来进行有条件的加载。最常见的条件注解有：

    • @ConditionalOnProperty：当某个配置属性满足特定条件时才生效。
    • @ConditionalOnMissingBean：当 Spring 上下文中不存在某个 Bean 时才生效。
    • @ConditionalOnBean：当 Spring 上下文中存在某个 Bean 时才生效。
    • @ConditionalOnClass：当类路径中存在某个类时才生效。

例如数据库或者redis的自动配置类，在有对应的配置属性才会自动装配。

@Configuration
@ConditionalOnClass(DataSource.class)
@EnableConfigurationProperties(DataSourceProperties.class)
@Import({DataSourcePoolMetadataProvidersConfiguration.class,
         DataSourceInitializationConfiguration.class})
public class DataSourceAutoConfiguration {
 
    @Bean
    @ConditionalOnMissingBean
    public DataSource dataSource(DataSourceProperties properties) {
        // 创建并返回 DataSource 对象
        return properties.initializeDataSourceBuilder().build();
    }
}

Important

关于spring.factories

spring.factories其实是SpringBoot提供的SPI机制,底层实现是基于SpringFactoriesLoader检索ClassLoader中所有jar引入的META-INF/spring.factories文件。基于注解将对应的实现类加载注册到IOC容器中。这种方式对于@ComponentScan不能扫描到的并且想自动注册到IOC容器的使用场景十分合适，基本上绝大多数第三方组件甚至部分spring-projects中编写的组件都是使用这种方案。

spring2.7之前都是使用这种默认方案，然而在springboot3.0有所变化

新的方案比较简单，就是在类路径下创建META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports文件，文件的内容是：每个实现类的全类名单独一行。例如对于使用了（低版本还没适配Spring Boot 3.0）mybatis-plus、dynamic-datasource组件的场景，可以在项目某个模块的resources目录下建立META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports文件，并写入对应实现类：

# ❌ 旧方式：META-INF/spring.factories
org.springframework.boot.autoconfigure.EnableAutoConfiguration=\
com.example.MyAutoConfiguration,\
com.example.AnotherAutoConfiguration

# ✅ 新方式：META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports
com.example.MyAutoConfiguration
com.example.AnotherAutoConfiguration
