1. 说说JVM的主要组成部分和流程

JVM主要包含两个子系统和两个组件，两个子系统分别是Class loader(类装载)、Execution engine(执行引擎)；两个组件分别是Runtime data area(运行时数据区)、Native Interface(本地接口)。

  • Class loader(类装载)：根据类的全限定名来装载class到运行时数据区域的方法区中
  • Runtime data area(运行时数据区)：即我们常说的JVM的内存
  • Execution engine(执行引擎)：负责将字节码文件“翻译”成对应平台上的机器指令并执行
  • Native Interface(本地接口)：和其它编程语言交互的接口

流程：编译器首先将Java代码转换成字节码。类加载器负责将字节码加载到运行时数据区域的方法区中。由于字节码是跨平台语言，因此需要解释器引擎根据平台翻译成具体的机器指令再交由cpu执行。本地库接口则是方面java代码进行系统调用或者其他函数的调用。

2. 讲讲JVM内存模型

  • 程序计数器（线程私有）：

    存放当前线程的下一条指令地址。通过改变它来实现程序的流转控制，如循环、跳转、线程恢复等

  • 虚拟机栈（线程私有）：

    主要存放栈帧；每个方法被执行时，都会同步创建一个栈帧，用于存储局部变量表、操作数栈、动态链接、方法出口等信息。方法从调用到执行完成，对应着一个栈帧在虚拟机栈中从入栈到出栈的过程。

  • 本地方法栈（线程私有）：

    和虚拟机栈功能类似。不同的是它是为Native方法服务的。

  • 堆（共享）：

    几乎所有的对象实例和数组都在这里分配内存，这里是GC的主要区域，也被称为GC堆

  • 方法区（共享）：

    存储内容包括已被加载的类的元数据信息（如类名、方法信息、字段信息）、类静态变量、运行时常量池和即时编译器编译后的代码缓存等

Important

部分名词概念：

局部变量表：包括基本数据类型和对象引用，其中基本数据类型是没有实例的概念的，这部分数据都在这里面

操作数栈：存储计算过程的中间结果和临时操作数

动态链接：栈帧内部的一个指针，指向运行时常量池（Runtime Constant Pool） 中该栈帧所属方法的引用。主要作用是将符号引用转为直接引用。其中**“动态”**的含义在于有些方法的直接引用需要到运行时才能确定。

方法出口：存放着调用该方法的指令的下一条指令的地址

常量池（Constant Pool Table）：class文件的一部分，用于存放编译器的各种字面量和符号引用，这部分内容在类加载后存放到方法区的运行时常量池中

字符串常量池（interned strings）：字符串常量池存储的是string对象的直接引用，真正的实例一直都在java堆中

Tip

实例常量（final）：不能被修改，和实例关联

静态变量(static)：和类关联在一起，随类的加载而加载，是类数据在逻辑上的一部分

全局常量(static final)：在编译的时候就会被赋值

3. Jva堆内存划分

Java堆 = 新生代+老年代 （通常是1:2）

新生代 = Eden + S0 + S1 (通常是8:1:1)

4. 说说JVM有哪些垃圾回收器

新生代垃圾回收器有：

  1. Serial（复制算法）：

    新生代单线程收集器，标记和清除过程都是单线程，简单高效。在进行GC时，必须暂停其他所有的工作线程直到GC结束。

  2. ParNew（复制算法）：

    新生代并行收集器，实际上是Serial的多线程版本，在多核环境下效果比Serial更好。在进行GC时，必须暂停其他所有的工作线程直到GC结束。

  3. Parallel Scavenge（复制算法）：

    新生代并行收集器，追求最高的吞吐量，尽可能减少总的GC时间，意味着不适用交互频繁的任务（如互联网需要较低的响应延迟），但适合科学计算等。在进行GC时，必须暂停其他所有的工作线程直到GC结束。

老年代垃圾回收器有：

  1. Serial Old（整理算法）：

    老年代单线程收集器，Serial收集器的老年版本。在进行GC时，必须暂停其他所有的工作线程直到GC结束。

  2. Parallel Old（整理算法）：

    老年代并行收集器，吞吐量优先，Parallel Scavenge的老年代版本。在进行GC时，必须暂停其他所有的工作线程直到GC结束。

  3. CMS（清除算法）：

    追求最短GC回收停顿时间，具备高并发、低停顿的特点。

面向全堆收集器：

  1. G1
  2. ZGC

5. 介绍一下CMS

CMS（Concurrent Mark-Sweep）：以追求最短回收停顿时间为目标的垃圾收集器（可能吞吐量会下降）。其名字暗号了核心特点

  • 并发：大部分垃圾收集工作线程可以与用户应用程序线程同时工作。
  • 标记清除：使用标记-清除算法进行垃圾回收工作。

CMS的整个工作过程可以分为以下4个阶段

  • 初始标记：
    只是标记一下GC Roots能直接关联的对象，速度很快，需要暂停工作线程
  • 并发标记：
    进行GC Roots跟踪的过程，和用户线程一起工作，不需要暂停工作线程
  • 重新标记：
    为了修正并发标记期间由于程序运行导致已有标记产生变动的一部分对象的标记记录，需要暂停工作线程
  • 并发清除：
    清除不可达对象，不需要暂停工作线程

由于耗时最长的并发标记和并发清除都是可以和用户线程一起工作，所以总体上来看CMS是和用户线程一起并发工作的。

6. 介绍一下G1(TODO)

相较于CMS不惜一切代价追求最短停顿时间，G1在保证可控的吞吐量的损失的前提下，提供更可预测的、更短的停顿时间。

它解决了CMS最致命的两个问题：

  • 内存碎片问题：G1使用标记整理算法，避免了清除算法导致的内存碎片
  • 不可预测的停顿：G1可以每次仅回收一部分区域，使停顿时间变得可预测和可控

G1收集器为了避免全区域GC，将堆内存划分为多个大小固定的独立区域，并且跟踪这些区域的垃圾收集进度，同时在后台维护一个优先级列表，每次根据所允许的收集时间，优先回收垃圾最多（回收利益最大）的区域。区域划分和优先级划分确保G1高效的垃圾收集效率。
这种划分是的G!每次不用回收整个堆空间，而是以增量的方式来处理，这也被成为增量式垃圾收集。

G1的特点：

  • 宏观上不分代：G1将宏观上的新生代老年代空间划分取消了，转而将堆空间划分为若干大小固定的区域（如2MB）。但是每个分区即可能扮演新生代也可能扮演老年代（确定的某个时刻只会是其中一种），分代的概念从物理意义上取消了，但仍然保留在逻辑意义上。同时除了新生代和老年代，还多设置了一个巨型对象区域
  • 使用标记整理算法：避免了内存碎片的产生
  • 分区回收：确保GC的高效（毕竟新生代往往垃圾最多）

G1提供了两种GC模式：

  • Young GC:

    当所有被指派为Eden角色的Region都被对象填满，无法再分配新对象。此时会触发Young GC。其他的和以前的Minor GC差不多。

  • Mixed GC:

    Mixed GC不仅进行正常的新生代垃圾收集工作，同时也回收部分后台扫描线程标记的老年代分区

7. 什么情况会触发FUll GC

  1. 系统调用：System.gc()或者Runtime.getRuntime().gc()

  2. 老年代空间不足

    • 晋升失败：新生代对象晋升到老年代
    • 分配失败：大对象大数组直接进入老年代

  3. 方法区耗尽

  4. 内存碎片严重

    例如CMS/G1等收集器由于使用整理算法导致内存碎片化严重，此时没有足够的连续空间分配大对象会触发Full GC

  5. 并发模式失败（CMS特有）

    场景：在CMS并发清除阶段，应用程序继续分配对象导致老年代被填满

    原因：CMS的并发清理阶段不会STW，如果在此期间老年代空间耗尽，就会退化为Serial Old收集器（让用户线程停止），触发Full GC。

8. 说说对象的分配规则

  1. 优先在Eden区分配

  2. 大对象直接进入老年代

    大对象：需要占用大量连续内存空间的对象，如长数组和大字符串

  3. 长期存活的对象晋升老年代

    除了达到设定年龄的直接晋升外，还有一个动态年龄判断

    动态年龄判断：如果某个年龄的X对象总大小 > Survivor空间的一半（也就是s1的一半），那么所有年龄大于这个年龄X的对象直接晋升

  4. 空间分配担保

    在触发Minor GC前，JVM会检查老年代剩余空间是否 > 历次晋升到老年代对象的平均大小

    • 如果足够并开启担保机制：则允许冒险尝试Minor GC
    • 如果空间不足：则可能直接触发Full GC

  5. 本地线程分配缓冲

    目的：减少多线程竞争堆内存分配时的同步开销

    原理：

    • 每个线程在Eden区拥有一小块私有内存（TLAB）
    • 对象优先在TLAB中分配，避免全局锁竞争。
    • TAB不足时，再通过共享的Eden区分配。

  6. 逃逸分析与栈上分配

    JVM通过分析对象的作用域，判断对象是否可能被外部方法或线程引用。对于未逃逸对象，可直接在栈上分配（无需进入堆），随方法调用结束自动销毁。

9. 说说类装载的执行过程

类装载分为以下 5 个步骤：

  • 加载：根据查找路径找到相应的 class 文件然后导入；
  • 验证：检查加载的 class 文件的正确性；
  • 准备：给类中的静态变量分配内存空间；
  • 解析：虚拟机将常量池中的符号引用替换成直接引用的过程。符号引用就理解为一个标示，而在直接引用直接指向内存中的地址；
  • 初始化：对静态变量和静态代码块执行初始化工作。

10. 双亲委派模型

  • 定义：

如果一个类加载器收到类加载的请求，它首先不会自己去尝试加载这个类，而是把这个请求委派给父类加载器完成。每个类加载器都是如此，只有当父加载器在自己的搜索范围内找不到指定的类时（即ClassNotFoundException），子加载器才会尝试自己去加载。

  • 作用：

在这里，先想一下，如果没有双亲委派，那么用户是不是可以自己定义一个java.lang.Object的同名类，并把它放到ClassPath中,那么类之间的比较结果及类的唯一性将无法保证，因此，为什么需要双亲委派模型？防止内存中出现多份同样的字节码。

  • 如何打破双亲委派：

    打破双亲委派机制则不仅要继承ClassLoader类，还要重写loadClass和findClass方法

Tip

例如Tomcat就需要打破双亲委派模型，通过为每个Web应用提供独立的WebAppClassLoader，并优先从应用私有目录加载类。

在web容器中，以下需求导致双亲委派模型不适用：

  • 隔离性：不同Web应用可能需要相同类库的不同版本（如Spring 4和Spring 5）
  • 灵活性：允许Web应用覆盖容器提供的类（如自定义的Servlet API实现）。如果使用双亲委派，可能会由父类加载器加载已有的类，从而导致自定义类不会被加载
  • 热部署：修改代码后无需重启即可重新加载类。

如果严格遵循双亲委派模型：

  • 所有Web应用共享父类加载器（如Common ClassLoader）加载的类，导致版本冲突。
  • 无法实现应用级别的类隔离。

13. 逃逸分析

逃逸分析的本质：主要就是分析对象的动态作用域，分析一个对象的动态作用域是否会逃逸出方法范围、或者线程范围。

逃逸分析的类型有两种：

  • 方法逃逸：对象是否会被方法外部的对象引用

    //StringBuffer对象发生了方法逃逸
public static StringBuffer createStringBuffer(String s1, String s2) {
    StringBuffer sb = new StringBuffer();
    sb.append(s1);
    sb.append(s2);
    return sb;
}

  • 线程逃逸：对象是否会被其他线程引用

如果能够证明一个对象不会逃逸到方法外或者线程外，或者说逃逸程度比较低，则可以对这个对象采用不同程度的优化：

  • 栈上分配：对于不会逃逸的对象，栈上分配可以快速在栈帧上创建和销毁对象，不用再将对象分配到堆空间，可以有效减少JVM垃圾回收压力

  • 标量替换：一个对象可能不需要作为一个连续的存储空间，也能被访问到，那么对象的部分可以不存储的在连续的内存，而是可以打散存储，甚至部分存储或者打散在CPU寄存器中。JVM将一个大的对象打散成若干变量的过程，也称之为分离对象

    比如下列代码：

    public static void main(String[] args) {
    alloc();
}
private static void alloc() {
    Point point = new Point（1,2）;
    System.out.println("point.x="+point.x+"; point.y="+point.y);
}
class Point{
    private int x;
    private int y;
}

    此时，JIT就会不会直接创建Point对象，而是直接使用两个标量int x,int y来替代Point对象,从而减少连续空间的消耗

  • 消除同步锁：如果同步块所使用的锁对象通过这种分析后，发现只能够被一个线程访问，根本用不着同步，那么，JIT编译器在编译这个同步块的时候就会取消对这部分代码的同步锁。所以：如果程序中使用了synchronized内置锁锁，则JVM会将synchronized内置锁消除。

    需要注意的是：这种情况针对的是synchronized锁，而对于非内置锁，比如 Lock 显示锁、CAS乐观锁等等，JVM并
    不能消除。

方法区相关

1. 永久代和元空间的联系和区别

方法区是虚拟的规范，是逻辑上的定义

永久代是JDK8之前的方法区实现，元空间是JDK8以及之后之后的方法区实现

核心区别如下（主要是位置区别）：

  • 永久代：位于jvm内存中，其大小收到堆容量的限制，其垃圾回收由Java堆垃圾回收机制管理（通常是Full GC）
  • 元空间：从Java内存中移除转而使用本地内存，Java堆垃圾回收机制不再负责其垃圾回收，但是其垃圾回收仍然由JVM负责

其他改进如下：

jdk7：将字符串常量池（interned strings）和类静态变量（class statics）转移到Java堆中

jdk8：元空间彻底取代永久代

Note

class对象一直在堆里面 类的元信息一直在方法区里面

jdk7以前，类静态变量在方法区中 jdk7之后类静态信息保存在堆里面
如果类静态变量是对象 对应的实例则一直在堆里面

2. 为什么要替换永久代？

  • 消除永久代的大小限制：永久代经常因为加载的类过多（尤其是在应用服务器、大量使用反射、动态代理的场景）而出现OutOfMemoryError，使用本地内存可以大幅减少这类问题。（永久代的调优通常是十分困难的）
  • 垃圾回收优化：永久代的垃圾回收通常和老年代的垃圾回收（通常是Full GC）捆绑在一起，而Full GC的代价非常高。元空间在本地内存可以使用高效、独立的垃圾回收
  • 未来改进优化：将HotSpot与JRockit虚拟机代码库合并（两者都是Oracle的），JRockit本来就没有永久代的概念
  • 元空间使用堆外内存：减轻GC压力；减少复制消耗（堆内在flush到远程时，会先复制到直接内存（非堆内存），然后在发送；而堆外内存相当于省略掉了这个工作。
  • ）

3. 类常量池、运行时常量池和字符串常量池

  • 类常量池：
    属于class文件的一部分，在编译期间，编译器将各种字面量和符号引用（如类和接口的全限定名）收集起来生成一个常量池
  • 运行时常量池：
    当JVM加载某个类的时候，会将该类的类常量池加载到方法区中，并将类常量池中的符号引用解析为直接引用（例如将某个方法解析为实际的方法代码在内存中的地址）
    同时运行时常量池还具备动态性（例如某个动态方法，或者是存储运行时产生的新常量）
  • 字符串常量池：
    隶属于运行时常量池的一个子集，专门用于存储字符串对象的直接引用来实现字符串的唯一性，从而节省内存并提升性能。

4. 方法区的垃圾回收

方法区的垃圾回收主要回收两部分内容：

  • 运行时常量池中废弃的常量（主要是字面量和符号引用）

    字面量比较接近Java语言层次的常量概念，如文本字符串、被声明为final的常量值等；而符号引用则属于编译原理方面的概念，包括下面三类常量：1）类和接口的全限定名；2）字段的名称和描述符；3）方法的名称和描述符。

    Hotspot虚拟机对运行时常量池的回收策略是很明确的，只要常量池中的常量没有被任何地方引用，就****可以被回收

  • 不再使用的类

    判定一个类型是否属于“不再被使用的类型”的条件比较苛刻，需要同时满足下面三个条件：

    • 该类所有的实例都已经被回收，也就是Java堆中不存在该类及其任何派生子类的实例。

    • 加载该类的类加载器已经被回收，这个条件除非是经过精心设计的可替换类加载器的场景，如OSGi、JSP的重加载等，否则通常是很难达成的。

    • 该类对应的java.lang.Class对象没有在任何地方被引用，无法在任何地方通过反射访问该类的方

      法。

1. 如何计算对象的内存占用 

public class MyOrder{
    private long orderId;
    private long userId;
    private byte state;
    private long createMillis;
}

Java对象在内存中由以下三部分组成：

  • 对象头
    • 标记字：8个字节（包含哈希码（HashCode）、GC分代年龄、锁状态标志、线程持有的锁、偏向线程ID、偏向时间戳）
    • 类型指针：8个字节（堆内存小于32GB默认开启指针压缩 仅占4个字节 因为对象都按照8字节对齐，即8B，那么4个字节即4G最大寻址32GB）
    • 数组长度（仅数组对象持有）：4字节
  • 实例数据
  • 对齐填充字节：保证对象总大小为8字节（64位系统，32位系统通常为4字节）的倍数

那么对象头大小为12字节，实例数据为25字节，合计37字节，根据对齐填充规则需要补充到40字节

2. 进阶：空String对象占用

String对象占用在不同jdk版本占用不同

在jdk8及之前，String构造如下：

public final class String
    implements java.io.Serializable, Comparable<String>, CharSequence {
    private final char value[];
    // Default to 0
    private int hash; 
}

此时String对象占用：

  • 对象头（12字节）
  • 实例数据（8字节）：value引用（4字节）+ hash（4字节）
  • 对齐填充：4字节

另外还包括value数组本身占用（16字节）：对齐（16）+ 填充（0）

因此合计为40字节

从jdk9开始，String引入了紧凑字符串优化，string内部不再用char[] 而是改用byte[]

这里以jdk17为例

public final class String
    implements java.io.Serializable, Comparable<String>, CharSequence,
               Constable, ConstantDesc {
    @Stable
    private final byte[] value;  
    //标识 value 数组中字节的编码格式 区分是占用1个字节编码 还是 2个字节编码（相当于jdk8之前的 也就是两个byte组成一个char这样）
    private final byte coder;             
    private int hash;
   //解决哈希码缓存中的歧义问题(因为哈希码缓存字段 hash 的默认值是 0)，即区分是没计算过 or 计算后的哈希码仍然是0
   private boolean hashIsZero;              
   static final boolean COMPACT_STRINGS;                  
}

和上面同理：
string对象占用12+4+1+4+1=20字节（COMPACT_STRINGS为静态变量 不属于对象）
value本身占用16字节
合计40字节

尽管最终答案一样 但是中间过程明显有所不同 需要注意

3. Java有哪些引用类型?

  • 强引用：gc不会回收
  • 软引用：在内存溢出前会被回收
    使用场景：可以用于一些不确定还用不用得到的缓存
  • 弱引用：下一次gc会被回收
    使用场景：当某个短生命周期的被长生命周期的对象临时引用 可以使用软引用避免前者一直无法回收
  • 虚引用：无法通过虚引用获取对象，虚引用的用途是在gc时返回一个通知。主要来跟踪对象被垃圾回收器回收的活动。虚引用能够正确判断对象是否被销毁回收。
