参考：计算机网络面试题 | 小林coding | Java面试学习

1. 浏览器打开一个网页，背后做了什么？

浏览器打开一个网页大致可以分为以下几个核心阶段：

  1. url解析

    浏览器会对url进行解析，确定协议、域名、端口号等信息

  2. dns域名解析

    浏览器需要根据域名解析对应的ip地址。

  3. 和服务器建立tcp连接

  4. 服务器处理请求并响应

  5. 浏览器接受响应并渲染客户端

  6. 连接关闭或者保持

2.Http不同版本的区别

Http/1.1的改进

Http/1.1相较于Http/1.0主要在连接效率和功能性上进行了改进：

  1. 持久连接

    • 问题：Http/1.0每次Http请求都要经历一次TCP连接的建立和断开，如果一个页面有多个资源，这种开销非常大

    • 改进：Http/1.1默认使用持久连接。在一个TCP连接上可以进行多次请求和响应，大大降低了网络开销

  2. 管道化

    • 问题：引入持久连接后，客户端需要等待上一个请求的响应回来后才能发起下一个请求（每次只能发送一个：队头阻塞）
    • 改进：引入管道化，允许客户端一次发送多个请求队列。客户端按顺序处理。但实际上由于响应顺序依赖，若前一个请求响应速度较慢，后续所有请求都会被阻塞，所以队头阻塞问题仍然存在

  3. 缓存机制增强

    • 改进：引入了更多、更精细的缓存控制头字段，如 Cache-Control、ETag、If-None-Match 等，提供了比 HTTP/1.0 的 Expires 更强大、更灵活的缓存策略。

  4. 支持分块传输编码、断点续传

  5. 新增了 OPTIONS, PUT, DELETE, TRACE, CONNECT 等方法。

Http/2.0的改进

  1. 二进制协议：Http/2.0使用二进制协议取代了Http1.x的文本协议

    • 问题：Http/1.x使用纯本文格式解析数据，效率低且容易出错，二进制更加高效

    • 改进：HTTP/2 将所有传输的信息分割为更小的消息和帧，并采用二进制格式对其进行编码。二进制格式解析更加高效，更节省带宽，并且减少了之前文本处理带来的各种安全问题和复杂性

  2. 多路复用

    • 问题：HTTP/1.1 的管道化未能彻底解决队头阻塞问题。浏览器为了规避这个问题，通常会为同一个域名开启多个 TCP 连接（通常是 6 个），但这又带来了额外的连接管理开销和竞争。
    • 改进：基于二进制分帧，HTTP/2 允许在同一个 TCP 连接中同时交错地发送多个请求和响应。请求和响应数据被拆分成多个帧，每个帧都有一个流ID（Stream ID）来标识它属于哪个请求/响应。请求方可以乱序传输，接收方根据流ID重新组装
    • 好处：彻底解决了 HTTP 层的队头阻塞问题。一个请求的延迟不会影响到其他请求。只需要一个 TCP 连接就能并行处理所有请求，极大提高了连接效率，避免了多个TCP连接的成本。

  3. 头部压缩：

    • 问题：HTTP/1.x 的头部（Header）是纯文本且不带状态，每次请求都会携带大量重复的、冗长的头部字段（如 User-Agent, Cookie, Accept 等），造成巨大的带宽浪费。
    • 改进：HTTP/2 使用 HPACK 压缩算法对头部进行压缩。同一连接上相同的头部字段只需传输一次，避免了 HTTP/1.x 中的冗余头部数据。

  4. 服务器推送：服务器可在客户端未请求资源时主动推送相关资源到客户端。适用于提前加载 CSS、JavaScript 等页面渲染所需资源，减少客户端加载时间。例如，当用户请求 HTML 页面时，服务器可以自动推送关联的 CSS 和 JavaScript 文件。

3. TCP/IP的三次握手和四次挥手

Tip

在理解具体过程前，需要先了解TCP报文中有几个重要的控制标志位：

  • SYN：同步序列号，用于发起一个新的连接
  • ACK：确认，用于确认收到的数据包，值为1表示有效
  • FIN：结束，用于释放一个连接

三次握手

三次握手过程（假设客户端请求服务端建立连接）：

  1. 第一次握手：

    客户端向服务端发送SYN包

    同时生成一个初始序列号seq=x

  2. 第二次握手

    服务端收到客户端的SYN包后，如果同意，会回复一个SYN+ACK包

    同时设置确认号ack=x+1，初始序列号seq=y

  3. 第三次握手

    客户端收到服务器的 SYN-ACK 报文后，必须再回复一个ACK确认包。

    同时设置确认号ack=y+1,序列号seq=x+1

至此，三次握手完成，双方都确认了对方的发送和接收能力正常，连接建立成功，可以开始传输数据。

Tip

为什么采用三次握手而不是两次握手？

目前主流说法是为了防止失效连接请求的干扰，考虑以下情况：

  1. 客户端向服务端发送SYN包请求建立连接，此时网络收到拥堵，服务端迟迟收不到请求
    1. 客户端认为之前的连接请求失效，向服务端重新发送SYN包
    2. 服务端收到并回应客户端SYN-ACK，两者建立请求
    3. 两者完成数据传输后，最后连接被关闭
  2. 此时之前的SYN包终于抵达服务端，服务器收到后会回应SYN-ACK，此时服务器认为连接已建立，并为此分配资源准备通信，但客户端认为连接已失效！

结果就是：服务器为这个“幽灵连接”白白分配了资源，并会等待一段时间。如果网络中存在大量这种过期的SYN包，服务器就可能被大量无效的半连接占满资源，导致拒绝服务（DoS），无法为正常请求提供服务。这是一种资源浪费和安全漏洞。

注意这里的漏洞更多是网络拥堵导致的！！！

Important

SYN-FLOOD

SYN-FLOOD是一种非常经典且危险的DDoS（分布式拒绝服务）攻击，它通过向目标服务器发送大量伪造的TCP连接请求（SYN包），消耗其资源，使其无法为正常用户提供服务的攻击方式，其攻击对象正是TCP协议的三次握手过程。

SYN—FLOOD攻击过程：

  1. 攻击者伪造大量的IP地址，并向目标服务器发送大量的SYN包
  2. 服务器收到每一个包，都会认为这是一个新的请求，并做出以下工作
    • 回复一个SYN-ACK包
    • 在内存中分配资源保存这个半连接的状态
    • 启动一个计时器，等待客户端的ACK回应
  3. 攻击者永远不会回复服务器
  4. 服务器会一直等待下去，直到超时（通常是30s到2分钟）才会 关闭连接

最终结果就是，服务器半连接队列被打满，无法再接受任何新的、合法的连接，导致拒绝服务。

应对措施：

  • 扩大半连接队列并缩短超时时间：面对恶意攻击作用不明显
  • 防火墙和DDos防护
    • 首包丢弃：设备故意丢弃第一个SYN包，如果对方是正常客户端，它会重传SYN，设备则认为其合法并放行。攻击程序通常不会重传。
    • 黑名单
    • 速率限制：对到达特定端口的SYN包数量进行限速。

四次挥手

TCP的连接的释放分为四步，因此成为四次挥手（其中前两次用于断开一方的连接，后两次用于断开另外一方的连接）

当客户端主动发起关闭是，工作流程如下：

  1. 第一次挥手：

    当客户端完成数据传输请求关闭连接时，客户端向服务端发送FIN包

    同时设置自身序列号seq=u（u为之前已经发送数据包的最后一个字节的序列号+1）

  2. 第二次挥手

    服务端收到FIN包后，会回复ACK包

    同时设置确认序列号ack=u+1, 自身序列号seq=v

    注意：此时从客户端到服务端的连接已经关闭，但是服务端可能还需要向客户端传输数据，此时的TCP连接状态称为半关闭状态

  3. 第三次挥手

    当服务端完成数据传输请求关闭连接时，服务端向客户端发送FIN+ACK包

    同时设置确认序列号ack=u+1,自身序列号seq=w（w为之前已经发送数据包的最后一个字节的序列号+1）

  4. 第四次挥手

    客户端收到请求后，回复ACK包

    同时设置确认序列号ack=w+1,自身序列号sql=u+1

    注意：

    • 客户端发送完ACK后，并不会立刻关闭，而是进入TIME-WAIT状态并等待一段长度为2MSL（Maximum Segment Lifetime，报文最大生存时间）的时间
    • 服务器一旦收到最终ACK，就会彻底关闭连接，释放资源
    • 客户端在等待完2MSL后也会彻底关闭连接

Important

为什么需要TIME-WAIT状态和2MSL的等待？

这是四次挥手中一个非常精妙且重要的设计。客户端在发送完最后一个 ACK 后必须等待 2MSL 的时间，主要有两个目的：

  • 可靠的终止连接：

    如果服务器发出的 FIN 报文（第三次挥手）丢失，那么服务器会超时重传这个FIN，再次发起第三次挥手。（客户端的ACK丢失也是一样，服务器也会重传）

    此时客户端就能收到重传的FIN并回复ACK，服务器才能够正常关闭资源

  • 让旧连接报文在网络中消逝

    MSL是一个报文能够在网络中存活的最大时间，等待2MSL可以让本次连接超生的所有旧报文从网络中消失，这样新建立的连接就不会被旧的连接影响，避免了数据错乱

4. HTTPS连接建立过程

为什么要用https？

因为http是明文传输的，存在数据泄露、数据篡改等风险，而https通过引入ssl/tls旨在解决这些问题。

SSL/TLS 在加密过程中涉及到了两种类型的加密方法：

  • 非对称加密：主要体现在会话密钥的生成过程
  • 对称加密：主要使用会话密钥对通信内容加密

Https连接建立步骤总体如下（以RSA密钥交换为例）：

  1. TCP三次握手：首先确立可靠的TCP连接

  2. TLS握手：TLS握手主要有四步

    1. 第一次握手，由客户端向服务器发起加密通信请求，也就是 ClientHello 请求，内容包含

      • 客户端支持的TLS版本
      • 客户端支持的密码套件列表，如RSA加密算法
      • 客户端产生的随机数n1

    2. 第二次握手，服务端收到客户端请求后，返回一条明文消息，包含：

      • 确认的TLS版本
      • 确认的密码套件列表
      • 服务器的数字证书（包含公钥、域名、颁发机构、有效期等信息）
      • 服务器产生的随机数n2

    3. 第三次握手，客户端会首先检查服务器数字证书的真实性，如果证书没问题，那么客户端就会从数字证书里面提取服务器的公钥，然后使用它加密报文，包含以下信息：

      • 客户端生成的随机数n3
      • 加密通信算法改变通知，表示后续的内容都将使用通信密钥加密通信
      • 服务器握手结束通知，表示服务器的握手阶段已经结束。这一项同时把之前所有内容的发生的数据做个摘要，用来供服务端校验。

      通信密钥：通过n1+n2+n3进行联合计算，各自生成本次https通信的[会话密钥]

    4. 第四次握手，服务端收到客户端的随机数n3后，同样根据协商的加密算法计算出[会话密钥]，然后想客户端发送：

      • 加密通信算法改变通知，表示随后的信息都将用「会话秘钥」加密通信
      • 服务器握手结束通知，表示服务器的握手阶段已经结束。这一项同时把之前所有内容的发生的数据做个摘要，用来供客户端校验。

  3. 发送加密的http请求：到这里开始使用http请求进行数据传输，只不过会使用会话密钥对内容加密

5. GET和POST区别

  1. 功能不同
    • get用于获取信息
    • post用于新增/修改信息
  2. 数据传递方式
    • get通过在url后面拼接参数
    • post是通过请求体进行数据传递，数据格式用Content-Type指定（multipart/form-data, application/json）
  3. 安全性

7. TCP和UDP的区别

特性 TCP UDP
连接性 面向连接 无连接
可靠性 可靠传输。使用确认、重传、拥塞控制等机制确保数据不丢失、不重复、不乱序。 不可靠传输。尽最大努力交付，但不保证数据一定到达。
数据形式 面向字节流。 面向报文。
首部开销 大（通常20-60字节）。包含序列号、确认号、窗口大小等复杂控制信息。 小（固定8字节）。只包含源/目标端口、长度和校验和。
传输速度 相对较慢。因为需要建立连接、确认、重传等额外开销。 非常快。没有建立连接和确认的开销，可以直接发送数据。
拥塞控制 有。能根据网络状况动态调整发送速率，避免网络瘫痪。 无。无论网络状况如何，都以恒定速率发送数据。
数据顺序 保证数据顺序。通过序列号对数据包进行排序。 不保证数据顺序。即使设置了序号，应用层也要自己处理乱序。
通信模式 只能点对点（一对一）通信。 支持一对一、一对多（多播）、多对多（广播）通信。
