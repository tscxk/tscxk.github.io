1.线程池参数

线程池工作原理：

线程池是为了减少频繁的线程创建和销毁带来的性能损耗。

当提交一个任务时，如果还有空闲的核心线程，那么就会将任务给核心线程
否则加入到等待队列，如果队列也满了，那么就会检测存活线程是否达到了最大线程
如果没有达到，那就创建新的线程并且从队列中获取队头的任务进行处理，然后将当前任务加入到等待队列末尾
如果存活线程已经到了最大线程数，就会按照线程池配置的拒绝策略处理任务

/*
* 线程池创建接口
**/
public ThreadPoolExecutor(
    int corePoolSize,
    int maximumPoolSize,
    long keepAliveTime,
    TimeUnit unit,
    BlockingQueue<Runnable> workQueue,
    ThreadFactory threadFactory,
    RejectedExecutionHandler handler) {
}
  • corePoolSize：核心线程数
  • maximumPoolSize：最大线程总数
  • keepAliveTime：存活时间。当线程池中线程数量大于核心线程，并且某个线程的空闲时间超过了keepAliveTime，那么这个线程就会被销毁
  • unit：存活时间单位
  • workQueue：工作队列
  • threadFactory：线程工厂。通常用来给线程取名字
  • handler：拒绝策略。当存活线程和工作队列都已经达到上限，对于新增任务按照拒绝策略进行处理
线程池有哪些拒绝策略？
  1. CallerRunsPolicy：将任务返回给调用者执行
  2. AbortPolicy：抛出任务拒绝的异常
  3. DiscardPolicy：将任务丢弃后不做任何处理（通常没人用）
  4. DiscardOldestPolicy：抛弃等待队列中最老的（队头）任务，然后将任务加入队尾（适用于新任务比旧任务重要的场景）
  5. 自定义拒绝策略：通过实现接口可以自定义任务拒绝策略。
线程池队列选择

  1. 有界队列（ArrayBlockingQueue）

    当任务数量稳定可控时，并且需要限制线程池中任务数量，防止任务挤压对系统造成过高压力时常采用有届队列

  2. 无界队列（LinkedBlockingQueue）

    当我们不关注任务是否挤压，即所有任务都需要处理，并且不存在任务积压风险是可以选择

  3. 同步队列（SynchronousQueue）

    SynchronousQueue是一个不缓存任何任务的队列，它会直接将任务交给线程池中的线程来执行。如果没有线程空闲，提交任务的线程将会被阻塞直到有线程来处理任务

    适用于需要快速处理每个任务的场景，常用于高并发、需要快速响应的场景

2. Volatile关键字

为什么要用volatile关键字？

  • 禁止指令重排序

    instance = new Singleton();并不是原子性操作，这里有三步

    • 分配内存空间
    • 初始化对象
    • 将对象引用赋值给instance

    如果没有volatile，那么可能会先执行第三步，那么拿到的对象就会是一个还没完全初始化的对象导致错误。

  • 保证变量的可见性

volatile如何保证变量的可见性

可见性问题的本质：每个线程都有自己的工作内存（缓存）。当线程读取变量时，可能从自己的工作内存中获取缓存数据，而不是从主存中获取数据，这就导致了可见性问题。同理，当更新数据时可能也是先对自己工作内存中的数据修改而没有立马同步到主存中也会导致类似问题。

可见性机制：

  1. 强制主存读写
    • 当写一个volatile变量时，JVM会立即将修改后的值刷新到主存中
    • 当读一个volatile变量时，jvm会直接从主存中进行读取
  2. 内存屏障
    • volatile通过在读写操作前后插入内存屏障来保证可见性
    • 写操作后的StoreLoad屏障会强制将写缓冲区的数据刷新到主内存
    • 读操作前的LoadLoad屏障会强制从主内存重新加载数据
哪些地方用到volatile
  • 双重检查锁
  • 和CAS搭配：例如Atomic类

3.HashMap为什么线程不安全

面试官：HashMap 为什么线程不安全？-腾讯云开发者社区-腾讯云

  1. 数据覆盖（JDK7/8）

    由于hashmap的put操作有以下几个操作：

    • 计算hash
    • 计算桶位置
    • 修改数据

    如果多个线程同时进行put操作，且桶位置计算结果相同，此时先修改的数据就会被后修改的覆盖掉，导致数据丢失

  2. 扩容导致死循环（JDK7）

    只需要AB扩容后还在一个桶内即可导致问题

    具体原因：头插法会导致仍然处于同一个桶位的元素倒序排序，例如有两个元素A->B始终在同一个桶位

    正常来说只有一个线程扩容最终会变成B->A的样子

    但是假如两个线程同时扩容就可能会导致A->B->A问题

  3. 扩容导致数据丢失（JDK7）

    只需要ACB 其中AB扩容后在一个桶内 C在另一个桶内

    当两个线程同时扩容时，会导致C死循环，B数据丢失

HashMap线程安全的解决方案：

  1. 使用HashTable（不推荐）

    HashTable的操作和HashMap操作基本一致，相当于在HashMap的所有方法上添加了Synchronized锁，性能极低

  2. 使用Collections.synchronizedMap（不推荐）

    对整个Map实例加了一个全局锁

  3. ConcurrentHashMap（推荐）

4. ConcurrentHashMap

总的来说，ConcurrentHashMap的线程安全体现在：

  1. 修改安全：CAS+Synchronized(jdk1.7使用分段锁)保证了数据修改不存在冲突，volatile保证了数据的可见性
  2. 扩容安全：使用全局的扩容table 同时逐个桶迁移
  3. 迭代安全
Jdk1.7

核心理念是分段。Jdk1.7采用分段锁机制，整体组成如下：

  • 一个ConcurrentHashMap由多个Segment组成（默认16个）。数量在创建时确定
  • 每个Segment：
    • 继承自ReentrantLock(可重入锁)，负责保护自己的分段
    • 内部维护了一个HashEntry数组，相当于一个小型HashMap
    • 都有自己的count（元素数量）和modCount（修改次数），用于统计和快速失败
JDK1.8

相较于jdk1.7主要有以下优化：

  1. 引进红黑树，当hash冲突的桶的链表长度达到8的时候会转为红黑树存储
  2. 不再是使用Segment锁，而是针对每个桶的头结点进行加锁单位。通常使用CAS+synchronized进行操作，避免不必要的锁开销。其中cas主要用于无锁操作（例如表初始化、桶为空的时候设置数据、对修改次数计数），sychronized通常用于竞争较为激烈的场景（例如桶结点数据不为空或者扩容操作）

5. ReentrantLock相关

ReentrantLock是java.util.concurrent包下的一个类，实现了Lock接口，并且通过名字可以发现其主要特性是可重入性。它的基础特性包含：

  • 可重入性：这意味着同一个线程可以多次获取锁而不会被阻塞。这和synchronized关键特性是一样的。
  • 显示锁：和synchronized的隐式锁（自动管理锁的获取和释放）不同，ReentrantLock锁的获取和释放需要手动控制。

基本使用示例如下：

public class Counter {
    private final ReentrantLock lock = new ReentrantLock();
    private int count = 0;

    public void increment() {
        lock.lock(); // 手动获取锁
        try {
            count++;
            // 其他需要同步的代码...
        } finally {
            lock.unlock(); // 必须在 finally 块中手动释放锁！
        }
    }
}

关键点：

  • lock()：获取锁。如果锁被其他线程获取，当前线程会被挂起，直到获取锁
  • unlock()：释放锁，需要注意的是一定要在finally块中手动释放

和Synchronized的区别

特性 Synchronized ReentrantLock
实现方式 java关键字，JVM实现 JDK级别的类，通过API实现
锁的获取 隐式获取和释放 显示调用lock()和unlock()
是否公平 仅支持非公平锁 支持公平锁和非公平锁
是否可中断 不可中断 ReentrantLock 可以响应中断，解决死锁的问题
用法 synchronized 可用来修饰普通方法、静态方法和代码块 ReentrantLock 只能用在代码块上

同时ReentrantLock还支持一些高级特性：

  • 尝试非阻塞获取锁
  • 可以为获取锁设置超时时间
  • 丰富的条件变量来实现复杂的同步逻辑，这是可重入锁的一个巨大优势

大部分时候我们都可以选择Synchronized，其代码更加简洁不容易出错，并且性能问题已经得到了很好的改善。

但是在以下需求产生时我们还是要考虑使用ReentrantLock：

  • 需要非阻塞、可中断、超时的高级锁特性
  • 需要公平锁特性
  • 需要实现复杂的同步逻辑，特别是需要多个条件变量的场景。

6. AQS相关

AQS是构建锁和同步器的基础框架，位于java.util的并发包中，jdk的许多同步工具都是基于AQS构建的，例如：

  • ReentrantLock

    例如ReentrantLock的Sync和UnfairSync都是继承自AQS

  • ReentrantReadWriteLock

  • Semaphore

  • CountDownLatch

核心机制：

  • state字段

    AQS 使用一个 int 类型的 volatile 变量（state） 来表示同步状态。这里state根据具体实现类的作用有不同的含义：例如在CountDownLatch中表示还需要倒数的数量，在Semaphore表示剩余许可证的数量，在ReentrantLock表示重入次数。

  • 变种CLH队列

    AQS 使用一个虚拟的双向队列（CLH 锁的变种）来管理等待线程

  • CAS操作

  • 预设的模板方法，通常我们只需要实现锁的获取和释放，内部逻辑都已经封装

7. CAS

CAS是并发编程中的重要基石，是构建许多高级并发工具的基础，它的全称是Compare And Swps（比较并交换），它通常包含三个操作数：

  • 内存位置（V）
  • 期望的原值（A）
  • 新值（B）

许多地方都会用到它，例如ReentrantLock、CountDownLatch、ConcurrentHashMap、Atomic等都会用到，它也是乐观锁的实现核心。

但是它也存在若干缺点：

  • ABA问题

    解决方案：添加版本号或者时间戳

  • 竞争激烈时循环时间长开销大

    解决方案：使用悲观锁，竞争激烈的时候不建议使用乐观锁

8. 为什么非公平锁吞吐量更大？

基础概念：

  • 公平锁：线程必须按照申请锁的时间排队获取锁
  • 非公平锁：允许“插队”，无论队列中是否存在线程排队，它都会尝试竞争一次锁，如果竞争失败，才会排队

非公平锁吞吐量更大的核心原因是因为它减少了线程切换的开销，也就是线程挂起和恢复的操作。

场景一

公平锁的工作模式：

  1. 线程A释放锁
  2. 操作系统唤醒等待队列的第一个被挂起的线程B
  3. 线程B被唤醒后，需要从内核态切换回用户态恢复然后尝试获取锁
  4. 线程C只能排队

非公平锁：

  1. 线程A释放锁
  2. 操作系统唤醒等待队列的第一个被挂起的线程B
  3. 此时来了一个线程C抢锁成功
  4. 线程B被唤醒后，发现锁已经被持有，继续挂起

可以发现线程C减少了一次线程切换的操作

场景二

对于一个刚刚释放锁又马上需要锁的线程（这种场景很常见），非公平锁的优势巨大。

如果这个线程能够连续获取锁，那么也能大大减少线程切换的开销

尽管非公平锁吞吐量更高，但是它也因此可能会导致线程饥饿

10. ThreadLocal基本原理

ThreadLocal是用于实现线程局部变量的类，它为每个线程都提供了独立的变量存储机制，实现了线程之间的数据隔离。

核心原理：

  • 线程隔离：线程之间互不干扰，相互隔离
  • 内部实现：每个Thread对象内部都维护了一个ThreadLocalMap，其中key为ThreadLocal本身，value为业务类型
  • 弱引用：ThreadLocal中的key使用弱引用指向ThreadLocal对象，避免内存泄漏

内存泄漏问题

ThreadLocal可能引起内存泄漏问题的原因：

  • key是弱引用，会在下一次GC回收，但value是强引用
  • 如何ThreadLocal实例被回收，但线程仍然存活，会导致value无法被访问也无法被回收

代码示例

public class ThreadLocalExample {
    public static void main(String[] args) {
        ThreadLocal<String> threadLocal = new ThreadLocal<>();
        
        threadLocal.set("Hello");
        System.out.println("Before GC: " + threadLocal.get()); // Hello
        
        // 线程仍然持有 ThreadLocalMap 的强引用
        // 但 threadLocal 变量本身的强引用仍然存在
        
        threadLocal = null; // 断开强引用
        
        System.gc(); // 触发 GC
        
        // 虽然 ThreadLocal 对象被回收，但线程中的 value 可能还在
        // 这就是内存泄漏的来源
    }
}

为什么key要设置为弱引用

在ThreadLocal对象被丢弃时，至少能自动清理key的引用，以及

正确用法

正确用法通常需要两个确保：

  • ThreadLocal使用static final修饰

    这样ThreadLocal始终存在且被强引用，那么作为ThreadLocalMap中的key也就不会被回收了（存在一个强引用，一个弱引用）

  • 使用remove及时清理线程变量

// 避免创建多个ThreadLocal
private static final ThreadLocal<User> USER_HOLDER = new ThreadLocal<>();

void handleRequest(Request req) {
    try {
        USER_HOLDER.set(getCurrentUser());
        // 处理请求...
    } finally {
        //一定要及时清理
        USER_HOLDER.remove();
    }
}

11.CopyOnWriteArrayList

CopyOnWriteArrayList是Java提供的一个线程安全的List实现，它主要通过“写时复制”来保证线程安全，适用于读多写少的场景。

核心思想：写时复制

顾名思义，“写时复制”表明在修改操作的时候，并不会直接对当前数组进行操作，而是先复制一份当前数组的副本，然后在副本上进行修改，修改完成后，再将原来数组的引用指向这个新的副本。

  • 读操作：直接读当前数据，无需加锁
  • 写操作：如add、remove会加锁，然后执行副本复制->修改副本->引用替换。并且锁是整个方法维度，这意味着修改操作仅支持一个线程并发，并不能支持多线程并发操作。

关键特性：

  • 读写分离
  • 线程安全：多个线程读取修改相互不影响
  • 弱一致性迭代器：其他线程修改并不会影响当前线程的迭代，当前线程如果在迭代过程，也不能即时看到修改后的数据
  • 写操作开销大：每次修改（包括修改单个元素）都需要复制整个底层数组，所以写操作的成本非常高。
  • 迭代过程不支持修改操作，如add、remove操作

12. CompletableFuture

CompletableFuture实现了Future和CompletionStage接口，提供了更为灵活的异步任务编排能力。相比传统的Future，它支持链式调用、异步任务组合和异常处理等能力，使得异步编程更加简洁高效。

Future的局限性

  • 获取结果会阻塞(Future 的目标是 “获取异步任务的结果”，在需要的时候会阻塞直到任务完成)
  • 无法链式组合多个任务
  • 缺乏异常处理机制
  • 无法手动完成任务

CompletableFuture的优势

  • 非阻塞获取结果（回调机制）
  • 任务链式组合
  • 异常处理
// 1. exceptionally（捕获异常并返回默认值）
CompletableFuture<String> future = CompletableFuture.supplyAsync(() -> {
    if (true) throw new RuntimeException("Error!");
    return "Success";
}).exceptionally(ex -> {
    System.err.println("Exception: " + ex.getMessage());
    return "Fallback Value";
});
System.out.println(future.get());  // 输出 "Fallback Value"

// 2.handle（无论成功或失败都处理）
CompletableFuture<String> future = CompletableFuture.supplyAsync(() -> {
    if (Math.random() > 0.5) return "Success";
    else throw new RuntimeException("Error!");
}).handle((result, ex) -> {
    if (ex != null) return "Fallback";
    else return result;
});
System.out.println(future.get());  // 可能输出 "Success" 或 "Fallback"
